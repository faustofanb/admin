## 2.1 安全目标与威胁模型

### 2.1.1 安全目标

系统安全目标从 CIA 三要素展开：

1. **机密性（Confidentiality）**
   - 不同租户之间的数据必须严格隔离。
   - 非授权用户无法访问不属于自己权限范围内的资源（用户信息、组织信息、资源文件等）。
   - 对敏感字段（密码、部分个人信息）进行加密/脱敏处理。
2. **完整性（Integrity）**
   - 所有变更操作（写接口）都必须经身份认证与权限校验。
   - 对关键数据（例如用户状态、角色权限、资源 ACL）的修改必须可追溯。
   - 防止中间人篡改请求、响应内容。
3. **可用性（Availability）**
   - 在被恶意请求、暴力尝试、部分组件故障的情况下，整体系统仍然保持可服务状态，而不是直接崩溃。
   - 对登录和高频操作接口设置限流与黑名单机制。
4. **可审计性（Accountability）**
   - 对关键操作（登录、权限变更、资源访问、任务执行）进行完整记录。
   - 审计记录不可轻易被篡改或删除，可以通过 Loki + PostgreSQL 双重存储进行比对。

### 2.1.2 威胁模型与防护面

典型威胁包含：

1. **多租户越权访问**
   - 攻击者通过伪造请求参数、修改 Header 中租户 ID、尝试访问其他租户的用户/资源数据。
   - 必须在所有数据访问层强制带上 `tenantId`，并在服务端从 `AppContext` 中获取，而非直接信任前端传参。
2. **水平越权 / 垂直越权**
   - 普通用户冒充更高权限用户（如管理员）访问管理接口。
   - 某用户访问同租户下其他用户的敏感信息。
   - 通过 RBAC +（必要时）ABAC 实现细粒授权；每个接口必须明确定义所需权限。
3. **身份伪造与重放攻击**
   - 窃取或伪造 JWT Token，冒充合法用户。
   - 拦截并重复发送重要操作请求，实现重放（特别是转账/资金/权限变更类操作，虽然当前系统没有资金场景，但要预留机制）。
4. **暴力破解与撞库**
   - 针对登录接口进行高频密码猜测。
   - 使用泄漏的密码库进行撞库攻击。
   - **网关层防护**：在网关层（Spring Cloud Gateway）统一识别恶意 IP 和高频请求，直接拦截，保护后端服务。
5. **数据泄漏**
   - 日志、监控输出敏感信息（密码明文、Token、秘钥等）。
   - 导出/Excel 功能泄漏不必要的信息字段。
6. **注入与脚本攻击**
   - SQL 注入：通过 ORM（Jimmer）+ 参数化查询防止。
   - XSS：前端主要防控，但后端应对输出字段进行合理转义与白名单控制（特别是富文本）。
7. **服务滥用与拒绝服务（DoS）**
   - 某租户或恶意 IP 对核心接口（登录、查询、上传）疯狂请求，拖垮系统。
   - 使用 Sentinel + Redisson 限流、熔断、隔离。

## 2.2 身份认证设计

### 2.2.1 账号模型

用户实体的相关字段：

- `tenantId`：租户 ID（逻辑隔离的关键）。
- `username`：用户名（在租户内唯一）。
- `email`（可选）：邮箱。
- `phone`（可选）：手机号。
- `passwordHash`：密码散列值。
- `status`：用户状态（正常、锁定、禁用、待激活）。

数据库约束（概要）：

```sql
-- 用户表中 (tenant_id, username) 联合唯一
UNIQUE (tenant_id, username)
```

### 2.2.2 密码策略

1. **密码存储**
   - 使用 BCrypt/Argon2 等强哈希算法。
   - 至少包含盐（BCrypt 内含盐），禁止明文或弱哈希（MD5/SHA1 等）。
2. **密码强度策略**
   - 最小长度配置（例如 ≥ 8 或 10 字符）。
   - 要求包含大写、小写、数字与特殊字符的组合（可配置）。
   - 可扩展：密码过期时间、历史密码禁止重复使用次数（由 `TenantConfig` 决定）。
3. **错误尝试与锁定**
   - 每账户在一定时间窗口内，密码连续错误达到阈值（例如 5 次），则锁定一段时间。
   - 锁定信息写入 user 状态 + 审计日志。

### 2.2.3 JWT 设计

JWT 用于访问令牌（Access Token），Refresh Token 可使用 JWT 或存储在 DB 中的随机 Token。

**Access Token 载荷建议字段：**

```json
{
  "sub": "u123", // 用户 ID
  "tid": "t1", // 租户 ID
  "roles": ["ADMIN"], // 角色列表
  "scopes": ["user:read", "user:write"],
  "iat": 1700000000, // 签发时间
  "exp": 1700001800, // 过期时间
  "jti": "uuid" // Token ID
}
```

- Token 签名使用对称密钥（HMAC）或非对称密钥（RSA/ECDSA）。
- 建议支持密钥轮换（kid 标记）：
  - `header.kid` 指定当前使用的密钥 ID。
  - 允许在配置中保留多把公钥/私钥对。

### 2.2.4 登录与刷新流程

1. **登录流程（Password Grant）概要：**
   - 客户端提交（POST `/api/v1/auth/login`）：
     - `tenantId`（或租户短名）
     - `username`
     - `password`
   - 服务端流程：
     1. 根据 `tenantId + username` 查用户。
     2. 校验密码哈希。
     3. 检查用户状态（禁用/锁定）。
     4. 计算并签发 AccessToken 与 RefreshToken。
     5. 记录登录审计（成功/失败）。
2. **刷新 Token 流程：**
   - 客户端提交（POST `/api/v1/auth/refresh`）：
     - RefreshToken。
   - 服务端流程：
     - 校验 Token 的有效性与是否被吊销。
     - 生成新的 AccessToken（与必要时新的 RefreshToken）。
     - 旧 AccessToken 不强制立刻失效，但可在黑名单机制中处理极端情况。
3. **登出流程（可选实现）：**
   - 若使用 RefreshToken + 黑名单方式，可以在登出时将当前 Token 标记为吊销（黑名单）。

## 2.3 授权与访问控制设计

### 2.3.1 RBAC 模型

RBAC 的基本关系如下：

```text
User --(多对多)--> Role --(多对多)--> PermissionMenu --(多对一)--> Permission
```

- 一个 User 属于一个 Tenant；
- 不同 Tenant 的 Role 与 Permission 按需复用或独立；
- 一个 Menu 可以绑定一个 Permission（如 `perm:sys:user:list`）。

### 2.3.2 Permission 表达式

为支持统一的权限控制，将 `Permission.expr` 定义为资源表达式：

例子：

- API 权限：`api:/api/v1/users:GET`、`api:/api/v1/users/**:POST`；
- 菜单权限：`menu:sys:user:list`；
- 资源权限：`resource:download`。

授权时，一般以 **权限码**（如 `perm:sys:user:list`）为主，对于复杂细粒度场景，可以基于资源表达式做匹配。

### 2.3.3 方法级授权（Spring Security）

对 Controller 和 Service 层采用注解方式控制访问：

```java
@PreAuthorize("hasAuthority('perm:sys:user:list')")
public Page<UserDTO> listUsers(UserQuery query) { ... }
```

- 权限数据在用户登录时从 DB 加载，放入 SecurityContext 与缓存中。
- 对于动态权限变化：通过 Outbox + MQ 通知权限变更，刷新用户权限缓存，必要时使相关 JWT 失效。

### 2.3.4 数据级授权（多租户与组织）

- **多租户层**：自动追加 `tenant_id = :tenantId`。
- **组织层（可选）**：
  - 若要求“用户只可看自己组织下的数据”，则在查询中增加 `org_path LIKE :orgPathPrefix` 过滤。

## 2.4 多租户隔离机制

### 2.4.1 AppContext 与上下文传播

AppContext 结构：

```java
public record AppContext(
    String tenantId,
    String userId,
    String requestId,
    String traceId) {}
```

- `tenantId`：从 JWT claims tid 获取；
- `userId`：从 JWT sub 获取；
- `requestId`：由网关或应用层生成，用于链路追踪；
- `traceId`：由 OpenTelemetry / tracing 框架生成。

### 2.4.2 REST Filter 实现（简要）

伪代码：

```java
public class AppContextFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) {
        try {
            String token = extractToken(request);
            Claims claims = jwtService.parseToken(token);
            AppContext ctx = new AppContext(
                claims.get("tid", String.class),
                claims.getSubject(),
                RequestIdGenerator.next(),
                TraceIdHolder.currentTraceId()
            );
            AppContextHolder.set(ctx);
            chain.doFilter(request, response);
        } finally {
            AppContextHolder.clear();
        }
    }
}
```

### 2.4.3 Repository 层强制多租户过滤

使用 Jimmer 时可以采用两种方式之一：

1. **全局过滤器**：在 Jimmer 配置中注册 `TenantFilter`，在所有查询自动追加 `tenant_id = currentTenantId` 条件。
2. **自定义 BaseRepository**：统一的 Query 方法自动带上租户条件。

> 目标是确保开发者不能轻易绕开多租户条件，即使忘记在业务代码中添加。

### 2.4.4 缓存与限流的多租户隔离

- Redis Key 一律带 `tenantId` 前缀：如 `t:{tenantId}:user:{id}`。
- Sentinel 热点参数规则：限定 `tenantId` 参数的访问频次，防止某租户拖垮整站。

## 2.5 传输安全与会话安全

### 2.5.1 HTTPS 与证书管理

- 所有外部访问接口统一使用 HTTPS；
- 由反向代理（例如 Nginx / Ingress Controller）终止 TLS；
- 定期轮换证书（Let’s Encrypt 或企业 CA）。

### 2.5.2 CORS 策略

- 仅允许受信任的前端域名；
- 对跨域请求应用适当的 `Access-Control-Allow-Origin` 策略；
- 严禁配置成 `*` 对所有源开放。

### 2.5.3 CSRF 防御

- 若使用 **JWT + Authorization Header** 方式（前端存储在 Memory/LocalStorage），则可以不启用 CSRF Token。
- 若使用 **Cookie 会话**，则必须启用 CSRF Token + SameSite 配置。

---

## 2.6 数据保护与隐私

### 2.6.1 敏感字段加密

- 数据库中敏感字段（如身份证号、银行卡号等）采用列级加密（可以使用加解密库进行透明封装）。
- 密钥托管通过 KMS 或独立密钥服务，而不是硬编码在代码中。

### 2.6.2 日志脱敏

- 日志中不得出现密码、原始 Token、银行卡号等。
- 对手机号、邮箱等进行部分脱敏（如 `138****1234`、`u***@mail.com`）。

### 2.6.3 数据导出与 Excel 脱敏

- 导出接口只允许导出与当前用户权限匹配的数据列。
- 后端严格控制字段子集，即使前端尝试通过参数请求更多字段也不应返回。

## 2.7 审计设计

### 2.7.1 审计事件范围

至少包括：

- 登录 / 登出 / 刷新 Token；
- 用户创建 / 删除 / 禁用 / 启用；
- 角色与权限变更；
- 资源上传 / 下载 / 删除；
- 调度任务创建 / 修改 / 手动执行；
- Excel 导入执行（包含成功/失败统计）。

### 2.7.2 审计表结构概要

```sql
CREATE TABLE t_audit_action (
  id          BIGSERIAL PRIMARY KEY,
  tenant_id   BIGINT NOT NULL,
  user_id     BIGINT,
  action      VARCHAR(128) NOT NULL,
  target_type VARCHAR(64),
  target_id   VARCHAR(128),
  detail      JSONB,
  ip          VARCHAR(64),
  ua          VARCHAR(255),
  ts          TIMESTAMP NOT NULL DEFAULT now()
);
```

- 与 Loki 日志形成“双记账本”：一份结构化在 PostgreSQL，一份在 Loki 中按 JSON 显示。

## 2.8 限流、防爆破、防滥用设计

### 2.8.1 登录防暴力破解

- 账号维度：在一定时间窗口内（如 5 分钟），同一账号登录失败超过 N 次，锁定该账号 M 分钟。
- IP 维度：统计某 IP 的登录失败次数，超过阈值后暂时拒绝其登录请求。
- Sentinel + Redis 结合统计与封禁。

### 2.8.2 关键接口限流

- 登录、注册、密码重置、文件上传、Excel 导入等接口要配置单独的限流规则：
  - Sentinel QPS 限制；
  - Redisson RateLimiter 控制租户维度速率。

## 2.9 安全事件响应与运维流程

### 2.9.1 安全告警来源

- Prometheus 告警（异常错误率、RT 突增）；
- Sentinel 熔断/限流命中率异常升高；
- 日志中出现异常的审计事件（例如大量登录失败、频繁权限变更）。

### 2.9.2 响应流程概述

1. 一线告警（告警平台） → on-call 工程师确认。
2. 判断是否为攻击或误操作：
   - 查询 Loki 日志与审计表，确认行为模式。
3. 必要时采取措施：
   - 降低某一租户的限流配额；
   - 暂时冻结部分账号；
   - 封禁异常 IP 段。
4. 分析根因并记录在案，修订防护策略或修复漏洞。

## 2.10 本章总结与对开发的约束

本章给出了从威胁模型到具体实践（认证、授权、多租户隔离、传输安全、数据保护、审计、限流与事件响应）的 **安全设计全景**。

对开发有以下硬性约束：

1. 业务代码不得自行从请求参数中读取 `tenantId`，必须从 `AppContext` 中取。
2. 所有写操作接口必须声明所需权限，并通过 Spring Security 校验。
3. 所有访问数据库的查询都必须包含 `tenant_id` 条件（由框架或基类保证）。
4. 日志中禁止记录密码、Token 明文以及敏感隐私字段。
5. 新增的对外 API 在设计时必须给出：
   - 必须权限；
   - 是否需要审计；
   - 是否需要限流。
