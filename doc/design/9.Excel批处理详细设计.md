## 9.1 设计目标与挑战

### 9.1.1 业务场景

后台管理系统常见的大批量数据操作：
- **导入**：用户导入 10 万+ 条用户数据、商品数据等。
- **导出**：财务报表、订单流水导出（可能百万级）。

### 9.1.2 核心挑战

1. **内存溢出（OOM）**：一次性加载大文件到内存会导致 JVM 崩溃。
2. **耗时过长**：HTTP 请求超时（通常网关限制 60s）。
3. **数据一致性**：导入一半失败如何回滚？部分成功如何反馈？
4. **并发控制**：多个租户同时导出大文件，CPU/IO 资源争抢。

### 9.1.3 解决方案选型

- **解析库**：**EasyExcel**（阿里开源），基于流式解析，内存占用极低。
- **处理模式**：**异步化**（上传 -> 提交任务 -> 轮询/通知结果）。
- **存储**：导入文件存 RustFS，导出结果存 RustFS。

## 9.2 总体流程设计

### 9.2.1 异步导入流程

1. **上传**：用户上传 Excel 文件，保存至 RustFS，获取 `resourceId`。
2. **提交**：调用 `POST /api/v1/excel/import`，提交 `resourceId` 和 `handlerType`（如 `USER_IMPORT`）。
3. **排队**：服务器创建 `ImportTask` 记录（状态 `PENDING`），发送消息到 MQ。
4. **处理**：
   - 消费者收到消息，下载 Excel 流。
   - 使用 EasyExcel 逐行读取。
   - 分批（Batch）写入数据库（每 1000 条一次）。
   - 记录成功/失败行数及错误详情。
5. **完成**：更新 Task 状态为 `FINISHED` 或 `PARTIAL_SUCCESS`，生成“错误报告 Excel”。

### 9.2.2 异步导出流程

1. **请求**：用户请求导出，带上查询条件。
2. **排队**：服务器创建 `ExportTask`，发送消息到 MQ。
3. **处理**：
   - 消费者分页查询数据库（流式或游标）。
   - 使用 EasyExcel 逐行写入本地临时文件或直接流式上传到 RustFS。
4. **交付**：更新 Task 状态，生成下载链接（`resourceId`）。
5. **通知**：用户在“任务中心”下载。

## 9.3 详细设计与实现

### 9.3.1 领域模型

**ImportTask**
- `id`, `tenantId`, `userId`
- `type`: `USER`, `ORDER`, ...
- `fileResourceId`: 原始文件
- `reportResourceId`: 错误报告文件（可选）
- `totalRows`, `successRows`, `failRows`
- `status`: `PENDING`, `RUNNING`, `FINISHED`, `FAILED`
- `startTime`, `endTime`

### 9.3.2 EasyExcel 监听器封装

为了复用“分批处理”、“错误记录”逻辑，封装通用 Listener：

```java
public abstract class BatchImportListener<T> extends AnalysisEventListener<T> {
    
    private final int batchSize;
    private final List<T> buffer = new ArrayList<>();
    private final List<ImportError> errors = new ArrayList<>();
    
    // 业务处理回调
    protected abstract void saveBatch(List<T> list);
    protected abstract void validate(T data, AnalysisContext context);

    @Override
    public void invoke(T data, AnalysisContext context) {
        try {
            validate(data, context);
            buffer.add(data);
            if (buffer.size() >= batchSize) {
                saveBatch(buffer);
                buffer.clear();
            }
        } catch (Exception e) {
            errors.add(new ImportError(context.readRowHolder().getRowIndex(), e.getMessage()));
        }
    }

    @Override
    public void doAfterAllAnalysed(AnalysisContext context) {
        if (!buffer.isEmpty()) {
            saveBatch(buffer);
        }
        // 处理 errors，生成报告...
    }
}
```

### 9.3.3 事务与回滚策略

大批量导入通常**不建议**整个文件一个大事务（长事务会导致数据库锁死、Undo Log 爆炸）。

**推荐策略：分批提交 + 错误容忍**
- 每 1000 条一个事务提交。
- 如果某批次失败，仅该批次回滚（或记录错误行）。
- 最终结果是“部分成功”。
- 提供“错误行报告”，用户修改后重新导入（仅导入失败的行）。

### 9.3.4 导出时的流式查询

避免 `findAll()` 加载所有数据到内存。

- **Jimmer/JPA**：使用 `Scroll` 或 `Stream` 接口。
- **MyBatis**：使用 `ResultHandler` 流式处理。

```java
// 伪代码：流式导出
excelWriter = EasyExcel.write(outputStream).build();
userService.streamAll(tenantId, userStream -> {
    userStream.forEach(user -> {
        excelWriter.write(convertToExportVO(user), writeSheet);
    });
});
excelWriter.finish();
```

## 9.4 性能优化与限流

### 9.4.1 并发控制

- **限制同时运行的任务数**：
  - 使用 Redis 计数器或 Sentinel。
  - 每个租户同时只能有 1 个导出任务，全系统最多 10 个。
- **线程池隔离**：
  - Excel 处理属于 CPU 密集型（解析）+ IO 密集型（DB/文件），应使用独立的线程池，避免阻塞 Web 线程。

### 9.4.2 虚拟线程的应用

- EasyExcel 的读取过程是阻塞的（InputStream read），非常适合虚拟线程。
- 数据库批量插入也是阻塞 IO。
- **结论**：Worker 端完全可以使用虚拟线程来执行导入导出任务，大幅提升吞吐量。

## 9.5 接口定义

### 9.5.1 提交导入

```http
POST /api/v1/excel/tasks/import
Content-Type: application/json

{
  "type": "USER_IMPORT",
  "fileResourceId": "123456789",
  "params": { "updateIfExists": true }
}
```

### 9.5.2 查询任务进度

```http
GET /api/v1/excel/tasks/{taskId}
```

响应：
```json
{
  "status": "RUNNING",
  "progress": 45, // 百分比
  "total": 10000,
  "processed": 4500
}
```

## 9.6 本章小结

本章设计了一套**异步、分批、流式**的 Excel 处理机制：
- 利用 **EasyExcel** 解决内存问题。
- 利用 **MQ + 异步任务** 解决超时与削峰问题。
- 利用 **分批事务** 解决长事务问题。
- 结合 **RustFS** 存储大文件。
