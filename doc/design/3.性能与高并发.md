## 3.1 性能目标与测试场景

### 3.1.1 性能设计总体目标

1. 在**多租户并发访问**场景下保持**稳定低延迟**。
2. 在**高 IO 压力**（数据库、Redis、MQ、RustFS）下充分利用 Java 21 虚拟线程能力。
3. 实现“**可度量、可诊断、可调优**”——所有关键路径都有指标和 Trace。

### 3.1.2 核心性能指标（初版）

> 后续在性能测试阶段会以此为基线进行验证和迭代。

- 用户登录接口
    - P95 < 200ms，P99 < 400ms（不含极端网络抖动）
- 常规业务查询（分页列表，例如用户列表）：
    - P95 < 150ms，P99 < 250ms（在单租户 1k 并发下）
- 资源上传（<10 MB）：
    - P95 < 800ms（完成 RustFS 写入 + 元数据落库）
- Excel 导入（1 万行以内）：
    - 整体处理时间 < 6 秒（含校验和入库）
- 运维任务（Outbox 扫描、资源清理）：
    - 单次任务执行时间可控在 1 分钟内，且不会明显拖累在线业务。

### 3.1.3 典型性能/并发场景

1. **多租户高并发登录**：
     - 50~200 个租户，平均每租户 10~100 并发用户随机登录。
     - 场景存在缓存命中和未命中两类请求。
2. **菜单/权限加载**：
     - 用户登录后或刷新时，加载当前用户的菜单树和权限列表。
     - 需考虑缓存命中率与重建菜单树的代价。
3. **资源上传/下载**：
     - 大量中小文件上传，每个请求涉及：
       - 鉴权 → 分布式锁（去重）→ RustFS 写文件 → DB 写 meta → 发布事件。
     - 下载需考虑热点资源访问导致的带宽与 IO 压力。
4. **Excel 导入/导出批处理**：
     - 大文件流式读取（读多，写多），校验逻辑复杂。
     - 多租户同时发起导入/导出操作。
5. **任务调度与异步事件处理**：
     - Outbox 扫描与 MQ 生产；
     - 定时报表与清理任务。

每个场景都应对应至少一个 Gatling/JMeter 测试脚本，并与 Prometheus 指标和 Trace 结合分析瓶颈。

## 3.2 虚拟线程使用策略

### 3.2.1 使用虚拟线程的原则

1. **所有 IO 密集型操作优先使用虚拟线程**，包括：
   - 数据库访问（Jimmer + PostgreSQL）；
   - Redis 操作（Redisson）；
   - RocketMQ 发送与消费逻辑；
   - 调用 RustFS 文件读写；
   - PowerJob Worker 内的处理任务；
   - Excel 导入的批量校验与写入。
2. **避免在虚拟线程内大量进行 CPU 密集计算**：
   - CPU 密集任务可分发到专门的线程池（如 ForkJoinPool），虚拟线程承担协调和轻量逻辑。
3. **避免 ThreadLocal 滥用**：
   - 虚拟线程与 ThreadLocal 能共存，但需要明确管理生命周期。
   - 对依赖 ThreadLocal 的框架（Seata、Spring Security、Dubbo、日志 MDC）要封装好“上下文传播”逻辑。

### 3.2.2 Spring Boot 虚拟线程配置

建议在 Spring Boot 配置中启用虚拟线程支持：

```yaml
spring:
  threads:
    virtual:
      enabled: true
```

同时可根据需要自定义执行器：

```java
@Bean
public ExecutorService virtualThreadExecutor() {
    return Executors.newThreadPerTaskExecutor(
        Thread.ofVirtual()
              .name("biz-vt-", 0)
              .factory()
    );
}
```

在控制器或服务中使用时：

```java
public void processAsyncTask(Runnable task) {
    virtualThreadExecutor.submit(task);
}
```

### 3.2.3 与上下文传播结合

为兼容 Seata、SecurityContext、AppContext 等 ThreadLocal，封装执行器：

```java
public final class ContextAwareVtExecutor {
    private final ExecutorService delegate;

    public ContextAwareVtExecutor(ExecutorService delegate) {
        this.delegate = delegate;
    }

    public void submitWithContext(Runnable task) {
        AppContext ctx = AppContextHolder.get();
        String xid = RootContext.getXID(); // Seata
        delegate.submit(() -> {
            try {
                if (ctx != null) AppContextHolder.set(ctx);
                if (xid != null) RootContext.bind(xid);
                task.run();
            } finally {
                if (xid != null) RootContext.unbind();
                AppContextHolder.clear();
            }
        });
    }
}
```

该模式避免了在虚拟线程环境下上下文丢失的问题。

## 3.3 数据库访问与读写分离

### 3.3.1 PostgreSQL 拓扑与读写分离策略

- **拓扑结构**：
  - 1 个主库（写 + 少量强一致读）；
  - 1~N 个从库（只读，用于查询与报表）。
- **读写路由**：
  - Command Handler、事务写操作 → 主库；
  - Query Handler、只读接口 → 从库（可按租户或负载分配）。

可通过自定义 RoutingDataSource 实现：

```java
public class ReadWriteRoutingDataSource extends AbstractRoutingDataSource {
    private static final ThreadLocal<Boolean> writeFlag = new ThreadLocal<>();

    public static void markWrite() {
        writeFlag.set(true);
    }

    public static void clear() {
        writeFlag.remove();
    }

    @Override
    protected Object determineCurrentLookupKey() {
        return Boolean.TRUE.equals(writeFlag.get()) ? "master" : "slave";
    }
}
```

在 Command Handler 开始时 `markWrite()`，执行完成后 `clear()`。

### 3.3.2 SQL 与索引优化策略

1. **所有业务表均含 `tenant_id`** 列并建立联合索引：`(tenant_id, business_key)`。
2. 对常用查询字段建立合适索引（例如用户表上的 username、资源表上的 sha256 等）。
3. 对分页场景：
   - 优先使用键集分页（基于 id 或 timestamp），避免 offset 深度分页。
   - 示例：`WHERE id > :lastId ORDER BY id ASC LIMIT 50`。
4. 尽量避免 N+1 问题：
   - 使用 Jimmer 的 fetch/关联加载能力；
   - 对批量业务逻辑，先一次性查出所需数据。

### 3.3.3 事务与锁对性能的影响控制

- 使用尽量小粒度和短时间的事务；
- 避免长事务占用连接与锁导致阻塞；
- 对热点表（如登录失败记录）可拆分或引入专门的统计方式（如 Redis）。

## 3.4 缓存策略与热点数据优化（概览）

> 详细缓存策略章节见第 5 章，这里从性能角度重点强调用途和原则。

### 3.4.1 缓存分层

1. **本地缓存（Caffeine）**
   - 适合“对单实例访问频繁且变更不那么敏感”的数据：
     - 系统配置、枚举字典；
     - 租户基本信息；
     - 菜单模板等。
   - 失效策略：TTL + LRU，容量控制。
2. **分布式缓存（Redis/Redisson）**
   - 适合跨实例共享、读多写少的数据：
     - 用户信息、权限列表；
     - 菜单树；
     - 资源元数据（ResourceMeta）。

### 3.4.2 缓存更新策略

- 采用 Cache-Aside 模式，结合 Outbox 事件做**异步刷新**：
  - 写操作完成后写出 Outbox 事件；
  - 由 MQ 消费端刷新/删除 Redis 中的对应 Key。
- 从性能角度：
  - 减少同步写缓存时间；
  - 请求处理主路径只负责“写 DB + 写 outbox”，异步刷新缓存。

## 3.5 并发控制与 Back-Pressure

### 3.5.1 分层限流与并发控制

1. **入口层（网关/Nginx）**
   - 基于 IP/QPS 的粗粒度限流；
     - 防止流量突刺直接把应用打挂。
2. **应用层（Sentinel + Redisson）**
   - 针对关键接口（登录、上传、Excel 导入）进行精细限流：
     - 按租户限流（ParamFlow 规则）；
     - 按用户/IP 限流。
3. **资源层（DB / Redis / RustFS）**
   - 控制并发连接和操作数：
     - 数据库连接池上限；
     - Redis 连接数量和操作超时；
     - RustFS 的并发上传/下载限制（必要时）。

### 3.5.2 Back-Pressure 与降级策略

- 当某个下游（例如 DB）响应变慢时：
  - Sentinel 可根据 RT 或错误率自动降级相关接口；
  - 调用方返回“系统繁忙，请稍后重试”，并记录对应的降级日志与指标。
- Excel 导入场景：
  - 当导入任务堆积过多时，系统应拒绝新的导入请求（提示稍后重试或排队信息）。

## 3.6 Excel 与批处理性能设计

### 3.6.1 流式处理与分批

- 使用 EasyExcel 的 **SAX 模式** 读取，按批次（如 1000 行）处理，以避免内存爆炸。
- 每批数据的处理逻辑：
  - 校验输入行（格式/业务规则）；
  - 识别错误行并记录；
  - 正确行批量写入 DB（使用批量 insert/update）。

### 3.6.2 并行处理与虚拟线程

- 每 1000 行为一批，使用虚拟线程并行处理多个批次：
  - 控制最大并行度（如同时最多处理 4~8 个批次），避免压垮 DB。
  - 通过 Redisson 信号量或 RateLimiter 控制并行批次数。
- 示例伪代码：

```java
Semaphore semaphore = new Semaphore(maxParallelBatches);
for (BatchData batch : allBatches) {
    semaphore.acquire();
    vtExecutor.submit(() -> {
        try {
            processBatch(batch);
        } finally {
            semaphore.release();
        }
    });
}
```

## 3.7 资源上传/下载性能设计

### 3.7.1 上传优化要点

1. **流式上传**，避免一次性将整个文件读入内存。
2. **分片上传（可选）**：对于大文件，支持分片并行上传，后端合并（可以按后续版本支持）。
3. **去重机制（秒传）**：
   - 在上传之前计算文件 sha256；
   - 查询是否存在相同 sha256 的已存资源；
   - 若存在则直接复用 storeKey，不再上传物理文件。

### 3.7.2 下载优化与热点防护

1. **开启 HTTP 缓存头（Cache-Control / ETag）**，对公共资源或弱敏感资源启用缓存。
2. **对热点资源应用限速措施**：
     - Sentinel 限流；
   - 若未来接入 CDN，可将热点内容缓存至 CDN 层。
3. **带宽与连接数监控**：
     - 通过 Prometheus + Grafana 监控下载流量、错误率与 RT，及时识别热点资源与带宽瓶颈。

## 3.8 任务调度与异步任务性能设计

### 3.8.1 PowerJob Worker 调度

- 所有 PowerJob 任务处理逻辑尽量使用虚拟线程，以提高任务并发能力。
- 对 CPU 密集型任务通过单独线程池控制并发，避免大量 CPU 计算占据虚拟线程。

### 3.8.2 异步任务对在线业务的影响控制

- 任务处理时：
  - 限制单台 Worker 并发任务数；
  - 通过 RateLimiter 或任务优先级控制资源占用；
  - 采取“离峰执行”策略（如报表任务在凌晨执行）。

## 3.9 性能监控与调优流程

### 3.9.1 监控指标分类

1. **系统层**：
   - CPU 使用率、内存占用、GC 次数与停顿时间。
2. **应用层**：
   - HTTP 请求数、LAT/P95/P99；
   - 线程/虚拟线程数；
   - DB 查询 RT 与 QPS；
   - Redis 操作 RT 与 QPS；
   - MQ 发送/消费延迟与失败率。
3. **业务层**：
   - 每租户的调用次数、错误率、限流次数；
   - 资源上传/下载成功率与耗时；
   - Excel 导入行数与耗时；
   - 任务调度执行成功/失败统计。

### 3.9.2 调优流程建议

1. 通过压测工具模拟典型场景。
2. 使用 Grafana + Trace 查看热点与瓶颈。
3. 针对发现的问题逐层优化：
   - SQL 优化、缓存策略调整、限流规则收紧或放宽；
   - 增加资源（副本数、节点数）或调整 JVM 参数；
   - 按租户/接口进行流量管理。

## 3.11 本章小结

本章从性能目标出发，结合：

- Java 21 虚拟线程；
- PostgreSQL 读写分离与 SQL 优化；
- 缓存与限流；
- 批处理（Excel）和资源上传下载；
- 调度任务与异步处理；

给出了一套围绕“**高并发、高 IO 效率、可调优**”的设计方案。

开发与架构实现时，应对照本章要求，确保：

1. IO 密集型操作均能迁移到虚拟线程环境；
2. 数据访问层实现读写分离并具备基础性能保障；
3. 关键路径有明确的缓存/限流与降级策略；
4. 所有性能目标均可在性能测试中被验证和监控。
