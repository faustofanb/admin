## 6.1 设计目标与边界

### 6.1.1 设计目标

资源模块（Resource Module）负责统一管理系统中的「资源」，目前主要是文件类对象（文档、图片等），未来可以扩展为更广义的“资源管理”：

- 支持多租户的资源隔离与访问控制（ACL）。
- 将资源**元数据**（结构化信息）与**实际内容**（文件数据）解耦：
  - 元数据存 PostgreSQL；
  - 内容存 **BlobStorage**（抽象层，支持 S3/MinIO/RustFS）。
- 支持高并发上传/下载，结合虚拟线程与限流、锁机制保证性能与稳定性。
- 提供统一的 REST API 和后续 RPC Facade 接口。
- 与 Outbox + MQ 集成，支持上传后的异步处理（缩略图、索引等）。
- 对访问进行审计和可观测（日志 + 指标 + Trace）。

### 6.1.2 边界与非目标

在当前版本中：

- 聚焦于**文件型资源**：包括文档、图片、PDF、压缩包等。
- **存储抽象**：不绑定特定厂商，通过 `BlobStorage` 接口适配 AWS S3, Aliyun OSS, MinIO 或自建 RustFS。

未来可以在此模块基础上扩展。

## 6.2 领域模型与生命周期

### 6.2.1 领域对象概览

主要领域对象：

- `Resource`（聚合根）
- `ResourceMeta`（实体）
- `AccessPolicy`（值对象）
- 逻辑状态机（状态枚举）

### 6.2.2 ResourceMeta 字段设计

核心字段（与数据库表 `t_resource_meta` 对应）：

- `id`：资源唯一 ID（主键，BIGSERIAL 或雪花 ID）。
- `tenantId`：租户 ID（多租户隔离必备）。
- `originalName`：原始文件名。
- `storeKey`：在 BlobStorage 中对应的存储键（路径或对象 ID）。
- `mime`：MIME 类型（content-type）。
- `size`：文件大小（字节）。
- `sha256`：文件内容哈希（用于去重与校验）。
- `aclType`：访问控制类型：0-private / 1-public / 2-role-based。
- `tags`：JSONB 标签数组，便于扩展分类和检索。
- `status`：资源状态（UPLOADING, ACTIVE, DELETED 等）。
- `createdAt`、`updatedAt`：时间戳。

表结构示例（扩展自前文）：

```sql
CREATE TABLE t_resource_meta (
  id            BIGSERIAL PRIMARY KEY,
  tenant_id     BIGINT       NOT NULL,
  original_name VARCHAR(255) NOT NULL,
  store_key     VARCHAR(255) NOT NULL,
  mime          VARCHAR(128),
  size          BIGINT       NOT NULL,
  sha256        CHAR(64)     NOT NULL,
  acl_type      SMALLINT     NOT NULL DEFAULT 0,  -- 0:private,1:public,2:role-based
  status        SMALLINT     NOT NULL DEFAULT 0,  -- 0:UPLOADING,1:ACTIVE,2:DELETED
  tags          JSONB        DEFAULT '[]',
  created_at    TIMESTAMP    NOT NULL DEFAULT now(),
  updated_at    TIMESTAMP    NOT NULL DEFAULT now(),
  UNIQUE (tenant_id, store_key),
  INDEX idx_res_tenant_sha (tenant_id, sha256),
  INDEX idx_res_tenant_status (tenant_id, status)
);
```

### 6.2.3 状态机（ResourceStatus）

资源状态枚举：

- `UPLOADING`：上传中（元数据已写入，内容可能还未完全写完）。
- `ACTIVE`：可正常访问。
- `DELETED`：已逻辑删除，不再对外可见。
- （可选）`PURGED`：已完成物理删除，仅用于审计记录。

状态迁移示例：

```text
(无)          --创建-->   UPLOADING
UPLOADING     --上传完成--> ACTIVE
UPLOADING     --失败/回滚--> (无) 或 标记失败（不保留）
ACTIVE        --逻辑删除--> DELETED
DELETED       --清理任务--> PURGED (物理删除)
```

## 6.3 模块结构与分层

### 6.3.1 模块包结构

假设模块根包为 `com.example.app.resource`：

```text
com.example.app.resource
  ├─ app
  │   ├─ command        # 上传、删除、更新 ACL 等命令处理
  │   └─ query          # 获取资源详情、列表等查询处理
  ├─ domain
  │   ├─ model          # Resource 聚合、值对象、枚举
  │   ├─ service        # 领域服务，例如去重策略、ACL 决策
  │   └─ event          # 资源相关领域事件（ResourceUploadedEvent 等）
  ├─ infra
  │   ├─ repository     # Jimmer Repository 接口与实现
  │   ├─ rustfs         # 对 RustFS 的客户端封装
  │   └─ mq             # 与 RocketMQ 的集成（如资源事件生产者）
  ├─ adapter
  │   ├─ rest           # REST Controller（对外 HTTP 接口）
  │   └─ rpc            # Dubbo Facade 实现（供内部分布式调用）
  └─ config             # 模块相关配置（Bean 定义、属性）
```

### 6.3.2 职责边界

- `app` 层负责：
  - 协调上传/删除命令的执行；
  - 调用领域对象和 RustFS 客户端；
  - 写 Outbox 与事件。
- `domain` 层负责：
  - 决定资源状态变迁；
  - 校验访问控制；
  - 生成领域事件对象。
- `infra` 层负责：
  - 落库、访问 RustFS 和 MQ；
  - 可能提供事务内与事务外的基础设施操作封装。
- `adapter` 层负责：
  - 暴露 REST API 与 RPC 接口；
  - 承担参数校验与 DTO/VO 转换。

## 6.4 上传流程设计

### 6.4.1 上传接口概览

REST 端点（示例）：

```http
POST /api/v1/resources/upload
Content-Type: multipart/form-data
Authorization: Bearer <AccessToken>

Form Fields:
- file: <binary>
- aclType: private/public/role-based
- tags: ["avatar","contract"]
```

响应：

```json
{
  "code": 0,
  "data": {
    "id": "r_123",
    "originalName": "a.png",
    "size": 102400,
    "mime": "image/png",
    "aclType": "private",
    "status": "ACTIVE"
  },
  "traceId": "..."
}
```

### 6.4.2 上传时序（简要）

1. 客户端上传文件（多部分表单）。
2. Controller 层解析文件与参数，构造 `UploadResourceCommand`。
3. Application Command Handler：
   - 计算文件 `sha256`；
   - 基于租户维度构造锁 Key，尝试获取 Redisson 分布式锁；
   - 检查是否已存在该 sha256 的资源（同租户或全局策略）；
   - 如存在 → 直接关联已有物理文件，插入新 meta（或直接复用? 根据策略）；
   - 如不存在：
     - 调用 RustFS 客户端写入文件流，获取 `storeKey`；
     - 写入 `t_resource_meta`（状态 `ACTIVE` 或先 `UPLOADING`，再更新为 `ACTIVE`）；
     - 写 Outbox（`ResourceUploadedEvent`）。
4. 返回资源信息。

### 6.4.3 SHA 去重与锁机制

**Why**：防止在高并发上传相同文件时出现重复存储与数据不一致。

1. 计算 sha256（在服务端完成，可以用流式计算，避免一次性读入内存）。
2. 获取分布式锁（`lock:{tenantId}:upload:{sha256}`）：
   - 若锁不可立即获取，说明已有上传在进行，直接返回“重复上传处理中”或稍后重试；
   - 若锁获取成功，则进行查重与上传。
3. 查重策略选项：
   - **租户内去重**：同一租户下 sha256 相同即视为相同资源。
   - **全局去重**：所有租户共享物理文件，只在 meta 里各自挂载。可以支持租户级策略配置。

## 6.5 下载与访问控制设计

### 6.5.1 下载接口概览

REST 端点（示例）：

```http
GET /api/v1/resources/{id}/download
Authorization: Bearer <AccessToken>
```

服务器行为：

1. 校验用户是否有访问该资源的权限（根据 ACL 和权限体系）。
2. 从 DB 获取 `ResourceMeta`，根据 `storeKey` 调用 RustFS 读取流。
3. 设置 HTTP 响应头（`Content-Type`, `Content-Length`, `Content-Disposition` 等）。
4. 以流形式返回文件内容。

### 6.5.2 ACL 规则

`aclType` 取值：

- `private`：仅资源所属租户内，且具备特定权限/角色的用户可访问。
- `public`：任何经过认证的用户（甚至匿名用户；需额外配置）可访问。
- `role-based`：仅特定角色的用户可访问（可在 meta 的 tags 或附加表中定义角色列表）。

访问控制流程：

1. 确定 `tenantId` 一致：
   - 若资源所属租户与当前 `AppContext.tenantId` 不同，直接拒绝（多租户隔离）。
2. 根据 `aclType` 调用领域服务 `AccessPolicyService` 做进一步校验：
   - private：检查用户权限，如 `perm:resource:read` 或更细粒的资源级授权；
   - role-based：检查用户是否拥有资源指定的角色；
   - public：放宽限制（但仍可更新审计日志）。

## 6.6 RustFS 集成设计

### 6.6.1 RustFS 通信协议

根据实际 RustFS 部署与接口，可能是：

- HTTP/REST 接口（例如 `PUT /files/{storeKey}`，`GET /files/{storeKey}`）；
- gRPC 接口；
- 自定义二进制协议。

本设计假设 **HTTP/REST** 形式，便于演示。

### 6.6.2 RustFS 客户端接口抽象

在 `infra.rustfs` 包中定义接口：

```java
public interface RustFsClient {
    /**
     * 上传一个流到 RustFS，返回 storeKey。
     */
    String upload(InputStream input, long size, String sha256, String contentType);

    /**
     * 读取文件内容为 InputStream。
     */
    InputStream download(String storeKey);

    /**
     * 逻辑删除或物理删除文件（视策略而定）。
     */
    void delete(String storeKey);
}
```

具体实现 `HttpRustFsClient` 使用 Spring WebClient 或 OkHttp 调用 RustFS 服务。

### 6.6.3 虚拟线程与 IO 结合

所有 RustFS 调用都在虚拟线程中执行：

```java
public ResourceId doUploadAndSave(UploadCommand cmd) {
    return vThreadExecutor.submit(() -> {
        // 在虚拟线程中执行上传与 DB 写入
    }).get();
}
```

或使用 `Thread.ofVirtual().start(...)` 包裹，注意上下文传播。

## 6.7 删除与清理策略

### 6.7.1 逻辑删除

REST 端点：

```http
DELETE /api/v1/resources/{id}
```

行为：

1. 校验权限（权限码如 `perm:resource:delete`），并检查租户一致。
2. 将资源 meta 的 `status` 置为 `DELETED`，同时写 Outbox（`ResourceDeletedEvent`）。
3. 根据业务需求：
   - 立即或延迟物理删除 RustFS 文件；
   - 或仅打标，保留一段时间以支持“回收站”功能。

### 6.7.2 物理删除与清理任务

由 PowerJob 触发定时任务：

1. 扫描 `t_resource_meta` 中 `status=DELETED` 且 `deleted_at` 超过 N 天的记录。
2. 调用 RustFS 客户端物理删除 `storeKey`。
3. 将记录状态更新为 `PURGED`（或直接删除记录）。

可选地，将清理操作写入审计日志与 Loki。

## 6.9 安全、审计与可观测性

### 6.9.1 安全点归纳

- 所有资源访问必须校验：
  - `tenantId` 一致；
  - ACL 权限；
  - 资源状态为 `ACTIVE`。
- 对于 public 资源，也应考虑：
  - 是否允许匿名访问；
  - 是否需要签名 URL（特别是敏感文件）。

### 6.9.2 审计记录

对于以下动作记录审计：

- 上传资源（记录 tenantId、userId、resourceId、size、sha256）。
- 下载资源（记录 resourceId、是否成功、IP、UA）。
- 删除资源。
- 更改 ACL。

审计写入：

- PostgreSQL 审计表 `t_audit_action`；
- Loki 日志（结构化 JSON）。

### 6.9.3 指标与 Trace

建议指标（Micrometer）：

- `resource_upload_total`
- `resource_upload_duration_seconds`
- `resource_download_total`
- `resource_download_duration_seconds`
- `resource_delete_total`
- `resource_rustfs_errors_total`

Trace：

- 上传/下载请求链路中包含调用 RustFS 的 span；
- 结合 `traceId` 与审计 ID 做全链路追踪。

## 6.10 性能与高并发考虑（与第 3 章呼应）

- 上传、下载路径完全 IO 密集，适合虚拟线程；
- 对 RustFS 的连接数、超时等参数要配置合理；
- 对大型文件建议采用分片上传方案（后续版本支持）；
- 对热点资源下载需结合 Redis 缓存元数据、Sentinel 限流，必要时引入 CDN。
