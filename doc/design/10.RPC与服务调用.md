## 10.1 设计目标与架构

### 10.1.1 为什么需要 RPC

虽然目前项目是**单体架构（Monolithic）**，但为了：
1. **模块解耦**：模块间通过接口（Interface）通信，而非直接依赖 Service 实现类。
2. **未来演进**：当某个模块（如资源服务、报表服务）负载过高时，可零代码修改直接拆分为微服务。
3. **协议优势**：Dubbo 协议（Triple/gRPC）比 HTTP 更高效，支持流式调用。

### 10.1.2 架构模式：In-JVM 优先

在当前阶段，所有模块部署在同一个 Spring Boot 进程中。
- **Dubbo 模式**：配置为 `scope="local"` 或利用 Dubbo 的自动 In-JVM 优化。
- **通信开销**：几乎为零（直接方法调用），无序列化/网络开销。
- **开发体验**：与微服务完全一致（定义接口 -> 引用接口）。

## 10.2 接口定义规范

### 10.2.1 模块结构

```text
admin-backend
 ├─ module-user
 │   ├─ rpc-api-user    # 纯接口、DTO、异常定义（可被其他模块依赖）
 │   └─ rpc-impl-user   # 接口实现（仅被主程序聚合）
 ├─ module-order
 │   ├─ rpc-api-order
 │   └─ rpc-impl-order
```

### 10.2.2 接口设计原则

1. **粗粒度**：避免频繁的细粒度调用（如 `getUserName`, `getUserAge`），应提供 `getUserInfo`。
2. **参数对象化**：入参使用 `UserQueryDTO`，出参使用 `UserDTO`，避免使用 `Map` 或 `JsonNode`。
3. **幂等性**：写接口（Create/Update/Delete）必须设计为幂等，方便重试。
4. **异常处理**：
   - 定义统一的 `RpcServiceException`。
   - 避免将本地事务异常直接抛给调用方，应封装为错误码。

### 10.2.3 代码示例

**API 定义 (`rpc-api-user`)**：

```java
public interface UserRpcService {
    // 根据 ID 查询用户
    UserDTO getUserById(Long tenantId, Long userId);
    
    // 批量查询
    List<UserDTO> listUsers(Long tenantId, List<Long> userIds);
}
```

**实现 (`rpc-impl-user`)**：

```java
@DubboService
public class UserRpcServiceImpl implements UserRpcService {
    
    @Resource
    private UserService userService;

    @Override
    public UserDTO getUserById(Long tenantId, Long userId) {
        // 转换 Domain Model -> DTO
        return UserMapper.toDTO(userService.find(tenantId, userId));
    }
}
```

**调用 (`module-order`)**：

```java
@Service
public class OrderService {
    
    @DubboReference(check = false)
    private UserRpcService userRpcService;

    public void createOrder(OrderCmd cmd) {
        // 像调用本地方法一样调用
        UserDTO user = userRpcService.getUserById(cmd.getTenantId(), cmd.getUserId());
        // ...
    }
}
```

## 10.3 序列化与协议

### 10.3.1 协议选择

- **Triple (基于 gRPC)**：
  - 推荐作为默认协议。
  - 支持流式调用（Stream），适合大文件传输或实时数据推送。
  - 跨语言支持好（如果未来有 Go/Rust 服务）。
- **Dubbo 协议**：
  - 传统的 TCP 长连接协议，性能极高，适合 Java 生态内部。

**配置建议**：
```yaml
dubbo:
  protocol:
    name: tri
    port: 50051
```

### 10.3.2 序列化

- **Hessian2**：Dubbo 默认，兼容性好，但对 Java 8+ 新日期类型支持需注意。
- **Protobuf**：性能最好，但需要编写 `.proto` 文件，开发成本高。
- **JSON (Jackson)**：通用性强，性能稍弱。

**当前策略**：使用 **Hessian2**（默认），并配置 Java 8 Time 序列化支持；或者直接使用 **JSON** 序列化以便于调试。

## 10.4 过滤器与上下文传递

### 10.4.1 隐式参数传递

利用 `RpcContext` 传递全局上下文：
- `tenantId`
- `userId`
- `traceId` (OpenTelemetry)
- `lang` (国际化)

### 10.4.2 自定义 Filter

实现 `org.apache.dubbo.rpc.Filter`：

```java
@Activate(group = {CommonConstants.PROVIDER, CommonConstants.CONSUMER})
public class ContextTransferFilter implements Filter {
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        if (RpcContext.getContext().isConsumerSide()) {
            // 消费端：从 AppContext 取出数据放入 Attachment
            RpcContext.getContext().setAttachment("tenantId", AppContext.getTenantId());
        } else {
            // 服务端：从 Attachment 取出数据放入 AppContext
            String tenantId = RpcContext.getContext().getAttachment("tenantId");
            AppContext.setTenantId(tenantId);
        }
        return invoker.invoke(invocation);
    }
}
```

## 10.5 容错与降级

即使是 In-JVM，也建议配置容错策略，为未来拆分做准备。

1. **Retries（重试）**：
   - 读接口：`retries=2`。
   - 写接口：`retries=0`（除非实现了幂等）。
2. **Timeout（超时）**：
   - 默认 3000ms。
   - 慢接口（如报表）单独配置 10000ms。
3. **Sentinel 集成**：
   - Dubbo 自动适配 Sentinel。
   - 可对 RPC 接口配置流控规则（QPS）和熔断规则（异常比例）。

## 10.6 本章小结

本章确立了 **Dubbo** 作为模块间通信的标准方式：
- 采用 **API 模块化** 设计，解耦实现。
- 利用 **In-JVM** 特性保持单体架构的高性能。
- 通过 **Filter** 机制统一处理租户上下文与 Trace。
- 为未来向 **微服务架构** 平滑迁移打下基础。
