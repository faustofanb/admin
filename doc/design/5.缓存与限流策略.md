## 5.1 设计目标与总体思路

### 5.1.1 设计目标

1. 提升读性能：
   通过本地缓存 + 分布式缓存减少数据库与外部系统压力。
2. 控制访问速率：
   防止单租户、单用户或单 IP 大量请求拖垮系统（尤其是登录、上传、Excel 导入）。
3. 确保可控的一致性：
   在可接受的业务范围内，保证“读多写少”场景下缓存的最终一致性。
4. 提供可观测与可调优能力：
   所有关键缓存命中率、限流命中次数等均可度量，并可在运行中动态更新规则。

### 5.1.2 总体思路

- 缓存分两层：
  - 本地缓存（Caffeine）：减少远程访问，优先缓存低变动配置。
  - 分布式缓存（Redis/Redisson）：跨实例共享，存放租户/用户/权限/资源等热点数据。
- 限流分多维：
  - 网关级限流（Spring Cloud Gateway）：IP 黑名单、全局 QPS 防护。
  - 接口级限流（API QPS/并发）。
  - 租户级限流（热点租户保护）。
  - 用户/IP 级限流（防暴力攻击与滥用）。
- 一致性优先采用：
  - 写 DB + 发送事务消息（或写 Outbox）
    - MQ 消费端刷新或删除缓存
      即 “源数据为准，缓存可重建”。

## 5.2 缓存设计总览

### 5.2.1 缓存分类

从逻辑用途划分为：

1. **配置类缓存**
   - 如：系统参数、枚举字典、租户级配置（登录策略、密码策略等）。
   - 更新频率极低，可长时间缓存。
2. **业务数据缓存**
   - 用户基本信息、角色/权限集合、菜单树。
   - 更新频率中等，对实时性要求较高，但可以接受秒级到分钟级延迟。
3. **资源元数据缓存**
   - 文件/图片等 ResourceMeta，访问频繁但写入较少。
   - 可以采用“读缓存 + 定期/事件刷新”的策略。
4. **会话与状态缓存**（如黑名单 Token、登录失败计数等）
   - 实时性较高，有 TTL 管控。

### 5.2.2 缓存层次结构

```text
[ 应用服务器实例 ]
   ├─ 本地缓存 (Caffeine)
   └─ 分布式缓存 (Redis via Redisson)

[ 持久层 ]
   ├─ PostgreSQL (主写从读)
   └─ 其他外部资源 (RustFS, MQ 等)
```

访问路径优先级：

1. 本地缓存（如果存在相关 Key 且未过期）
2. Redis
3. DB 或外部系统

## 5.3 缓存键空间与命名规范

### 5.3.1 Key 命名原则

1. **所有业务缓存 Key 必须显式包含租户维度**（否则即不是租户相关数据或设计错误）。
2. Key 语义清晰，便于排查和运维。
3. 通过统一的工具类生成 Key，避免手写字符串。

### 5.3.2 Key 模板约定

建议统一围绕：

```text
t:{tenantId}:{domain}:{entity}:{identifier}
```

示例：

- 用户详情：
  `t:{tenantId}:user:detail:{userId}`
- 用户权限集合：
  `t:{tenantId}:user:perms:{userId}`
- 角色权限集合：
  `t:{tenantId}:role:perms:{roleCode}`
- 菜单树（按用户视角）：
  `t:{tenantId}:user:menus:{userId}`
- 资源元数据：
  `t:{tenantId}:resource:meta:{resourceId}`
- 租户配置：
  `t:{tenantId}:config:tenant_info`

### 5.3.3 Key 工具类示例

```java
public final class CacheKeys {
    private CacheKeys() {}

    public static String userDetail(String tenantId, String userId) {
        return "t:" + tenantId + ":user:detail:" + userId;
    }

    public static String userPerms(String tenantId, String userId) {
        return "t:" + tenantId + ":user:perms:" + userId;
    }

    public static String rolePerms(String tenantId, String roleCode) {
        return "t:" + tenantId + ":role:perms:" + roleCode;
    }

    public static String userMenus(String tenantId, String userId) {
        return "t:" + tenantId + ":user:menus:" + userId;
    }

    public static String resourceMeta(String tenantId, String resourceId) {
        return "t:" + tenantId + ":resource:meta:" + resourceId;
    }

    public static String tenantConfig(String tenantId) {
        return "t:" + tenantId + ":config:tenant_info";
    }
}
```

## 5.4 本地缓存（Caffeine）设计

### 5.4.1 适用场景

无需跨实例共享，读多写极少的配置类数据：

- 全局配置（如系统开关、阈值）；
- 枚举/常量字典；
- 少量不经常变的租户配置信息（可设置较短 TTL 防止“僵尸数据”）。

### 5.4.2 示例配置

```java
@Bean
public Cache<String, Object> localCache() {
    return Caffeine.newBuilder()
            .initialCapacity(1000)
            .maximumSize(10_000)
            .expireAfterWrite(Duration.ofMinutes(10))
            .recordStats()
            .build();
}
```

### 5.4.3 使用方式

典型访问模式：

```java
public TenantConfig getTenantConfig(String tenantId) {
    String key = "tenant:config:" + tenantId;
    return (TenantConfig) localCache.get(key, k -> loadTenantConfigFromDb(tenantId));
}
```

注意：

- 本地缓存仅作为性能优化，不参与一致性核心逻辑设计。
- 容量限制和 TTL 控制防止内存过大。

## 5.5 分布式缓存（Redis/Redisson）设计

### 5.5.1 Redisson 客户端配置

```java
@Bean(destroyMethod = "shutdown")
public RedissonClient redissonClient(@Value("${redis.url}") String url) {
    Config config = new Config();
    config.useSingleServer()
          .setAddress(url)
          .setConnectionMinimumIdleSize(8)
          .setConnectionPoolSize(64);
    return Redisson.create(config);
}
```

可根据环境使用：

- 单机：`redis://host:6379`
- 哨兵：`useSentinelServers()`
- 集群：`useClusterServers()`

### 5.5.2 核心使用模式

1. 简单 KV 缓存（RBucket）：

```java
public <T> void cacheValue(String key, T value, long ttlSeconds) {
    RBucket<T> bucket = redissonClient.getBucket(key);
    bucket.set(value, ttlSeconds, TimeUnit.SECONDS);
}

public <T> T getCachedValue(String key) {
    RBucket<T> bucket = redissonClient.getBucket(key);
    return bucket.get();
}
```

2. Hash 缓存（RMap）：适用于需要按字段更新的聚合数据，例如某些统计信息。

## 5.6 缓存一致性与更新策略

### 5.6.1 一致性级别选择

对不同数据类型采用不同一致性要求：

| 数据类型      | 一致性要求        | 方案                           |
| ------------- | ----------------- | ------------------------------ |
| 核心写模型    | 强一致（DB 为准） | 本地事务 + DB                  |
| 缓存内容      | 最终一致          | Outbox + MQ + 缓存刷新         |
| 会话/短期状态 | 近实时一致        | 直接写 Redis + TTL，失败即降级 |

### 5.6.2 Cache Aside 模式（读侧）

典型读流程：

1. 先查缓存：
   - 命中 → 返回；
   - 未命中 → 访问 DB，加载数据 → 回写缓存 → 返回。
2. 对“空数据”也可短暂缓存，防止穿透。

示例：

```java
public UserDTO getUser(String tenantId, String userId) {
    String key = CacheKeys.userDetail(tenantId, userId);
    UserDTO cached = getCachedValue(key);
    if (cached != null) return cached;

    User user = userRepository.findById(tenantId, userId)
                  .orElseThrow(NotFoundException::new);
    UserDTO dto = mapper.toDto(user);
    cacheValue(key, dto, 600);
    return dto;
}
```

### 5.6.3 Cache Invalidation（写侧）

写操作的原则：

- 不强制同步更新缓存，而是：
  - 写 DB；
  - 写 Outbox；
  - 由 MQ 消费端异步删除/刷新缓存。

优势：

- 写路径简单、延迟小；
- 更容易保持缓存一致性逻辑集中（在消费者侧）。

示例（消费端刷新缓存）：

```java
public void onUserUpdated(UserUpdatedEvent event) {
    String tenantId = event.tenantId();
    String userId   = event.userId();
    // 简单策略：直接删除缓存，下一次访问时回源加载
    String key = CacheKeys.userDetail(tenantId, userId);
    redissonClient.getBucket(key).delete();
}
```

### 5.6.4 缓存穿透、击穿、雪崩防护

1. **穿透**：大量请求访问不存在的 Key
   - 缓存“空值”短时间（例如 30 秒），避免每次都访问 DB。
2. **击穿**：某个热点 Key 失效的瞬间被高并发访问
   - 加分布式锁或单机互斥加载（如 `RLock` 或 Caffeine `get(key, loader)`）。
3. **雪崩**：大量 Key 同时过期
   - TTL 加随机偏移；
   - 拥有多层缓存（本地 + Redis）；
   - 使用限流保护 DB。

## 5.7 分布式锁与并发保护（Redisson）

### 5.7.1 典型使用场景

- 文件上传去重（根据 sha256 判重）；
- 资源元数据更新的串行处理；
- 任务并发控制（防止同一任务多次同时执行）；
- Excel 导入任务同一租户单线程执行。

### 5.7.2 使用示例：资源上传去重

```java
public ResourceId uploadResource(UploadCommand cmd) {
    String tenantId = cmd.tenantId();
    String sha256 = cmd.sha256();
    String lockKey = "lock:" + tenantId + ":upload:" + sha256;

    RLock lock = redissonClient.getLock(lockKey);
    // 尝试立即获取锁，避免阻塞主线程太久
    if (!lock.tryLock()) {
        throw new DuplicateUploadInProgressException();
    }
    try {
        Optional<ResourceMeta> existed = resourceRepository.findByTenantAndSha(tenantId, sha256);
        if (existed.isPresent()) {
            // 秒传逻辑：直接返回已有资源 ID
            return existed.get().id();
        }
        // 正常上传流程：写 RustFS + 写 DB + 写 Outbox
        return doUploadAndSave(cmd);
    } finally {
        lock.unlock();
    }
}
```

## 5.8 限流与熔断策略（Sentinel 视角）

> 这里侧重“设计与规则”，具体 Sentinel 配置细节可放在单独配置文档。

### 5.8.1 资源命名规范

为便于监控与管理，将 Sentinel 资源名规范化：

- HTTP 接口：`api:/method/path`
  - 如：`api:GET:/api/v1/users`
  - 如：`api:POST:/api/v1/resources/upload`
- 业务流程：`flow:{flowCode}`
  - 如：`flow:tenantOnboarding`
- 流程节点：`flow-node:{nodeId}`
  - 如：`flow-node:initDefaultRoles`

### 5.8.2 规则类型使用说明

1. **FlowRule（流控）**
   - 基于 QPS 或并发数对资源做限流。
   - 示例：
     - `api:POST:/api/v1/resources/upload` 每实例 QPS ≤ 100。
2. **DegradeRule（降级）**
   - 根据 RT 或异常比例/数触发熔断。
   - 示例：
     - `resourceUpload` 当 P95 RT > 1500ms 且持续 10s，则在接下来 30s 内快速失败。
3. **ParamFlowRule（热点参数规则）**
   - 对某一参数（如 `tenantId`）做精细限流，防止某单个租户过多占用资源。
   - 示例：
     - `api:GET:/api/v1/users`，对 `tenantId` 维度做热点限流，每租户 QPS ≤ 50。

### 5.8.3 接口限流与降级示例

在 Controller 或 Service 方法上标注：

```java
@SentinelResource(
    value = "api:POST:/api/v1/excel/import",
    blockHandler = "blockExcelImport",
    fallback = "fallbackExcelImport")
public ImportTaskDTO importExcel(MultipartFile file) {
    // 实际导入逻辑
}

public ImportTaskDTO blockExcelImport(MultipartFile file, BlockException ex) {
    throw new TooManyRequestsException("Excel 导入频率过高，请稍后再试");
}

public ImportTaskDTO fallbackExcelImport(MultipartFile file, Throwable ex) {
    // 出现非限流/熔断异常时的兜底逻辑
    throw new SystemBusyException("系统繁忙，请稍后重试");
}
```

## 5.9 Excel、资源等特殊场景的限流

### 5.9.1 Excel 导入

- 通过 Redisson RateLimiter 限制**每租户每分钟的导入次数**：

```java
RRateLimiter limiter = redissonClient.getRateLimiter("excel:import:" + tenantId);
limiter.trySetRate(RateType.OVERALL, 5, 1, RateIntervalUnit.MINUTES);
if (!limiter.tryAcquire()) {
    throw new TooManyRequestsException("当前租户 Excel 导入次数过多，请稍后再试");
}
```

- Sentinel 在接口级再做全局限流和熔断，防止高峰时段压垮系统。

### 5.9.2 资源上传/下载

- 资源上传：
  - 使用 Sentinel 限制 `api:POST:/api/v1/resources/upload` 的总 QPS。
  - 对单租户用 ParamFlowRule 做分流。
- 资源下载：
  - 针对公共资源可以提高阈值；
  - 对私有/敏感资源增加严格的访问日志记录（PLG + 审计表）。

## 5.10 监控、统计与调优

### 5.10.1 缓存相关指标

通过 Micrometer+Prometheus 记录：

- `cache_hits_total{cache="redis", domain="user"}`
- `cache_misses_total{cache="redis", domain="user"}`
- `cache_hit_ratio{cache="redis", domain="user"}`
- `cache_rebuild_latency_seconds{domain="resource"}`

### 5.10.2 限流/熔断相关指标

- `sentinel_block_total{resource="api:POST:/api/v1/excel/import"}`
- `sentinel_degrade_total{resource="resourceUpload"}`

结合告警规则：

- 当某接口的 block 率 > 30% 且持续 5 分钟，视为“限流过严”，需要运维调整阈值或扩容。

## 5.12 本章小结

本章从“**缓存**”与“**限流**”两个关键维度，给出了：

- Key 设计与多租户前缀规范；
- Caffeine + Redis/Redisson 的两级缓存策略；- 一致性保障方式（Cache Aside + Outbox + MQ）；
- 分布式锁与 RateLimiter 在资源上传/Excel 导入等场景的使用；
- Sentinel 流控、熔断、热点参数限流的规则设计；
- 缓存/限流指标与调优手段。

在实际开发中：

- 所有需要缓存的查询接口应先明确：缓存层级、TTL、Key 规范；
- 所有需要限流的接口应先明确：限流维度（系统/租户/IP）、阈值与降级方式；
- 缓存更新应统一通过事件驱动方式进行，避免各个业务模块自行操作缓存造成混乱。
