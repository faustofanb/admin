# 关于 System 模块架构模式的决策记录

## 1. 背景

项目 `backend-core` 模块中提供了 CQRS（命令查询职责分离）和 Event（领域事件）的基础支持，但在 `backend-system` 模块的初始实现中，采用了传统的 **Controller -> Service -> Repository -> Entity** 分层架构。本文档旨在记录这一架构决策的考量因素及未来的演进策略。

## 2. 现状分析：为何选择分层架构？

目前 `backend-system` 采用经典分层架构，主要基于以下考量：

### 2.1 业务属性适配

`backend-system` 模块包含大量基础管理功能，如：

- 字典管理 (`SysDict`)
- 参数配置 (`SysConfig`)
- 基础用户管理 (`SysUser`)
- 日志与文件管理

这些功能本质上是 **CRUD (增删改查)** 操作，业务逻辑相对单一。对于此类“管理后台”性质的功能，分层架构是最简单、最高效的开发模式。

### 2.2 框架特性契合 (Jimmer)

项目选用的 ORM 框架 **Jimmer** 具有以下特性：

- **实体即接口**：Jimmer 的实体定义为 Interface。
- **强大的数据抓取 (Fetcher)**：Jimmer 擅长处理任意形状的数据图谱查询。
- **动态 SQL 支持**：天然倾向于让开发者关注数据的获取与组装。

这种特性使得“贫血模型 + 强力查询”的模式在 Jimmer 下开发体验极佳，效率远高于强制封装领域行为。

### 2.3 开发阶段权衡

项目目前处于 **Foundation (基础设施)** 和 **Setup (模块初始化)** 阶段。首要目标是快速建立基础设施、跑通全链路流程。过早引入 DDD 的战术模式（如聚合根、值对象、工厂、复杂的 Repository 封装）会显著增加样板代码和认知复杂度，拖慢初期建设速度。

## 3. 演进策略：是否引入 DDD？

**结论：采用“轻量级 DDD”或“混合模式”，拒绝教条式的全盘 DDD 化。**

### 3.1 保持现状的部分 (CRUD 核心)

对于以下模块，继续保持简单的分层架构，避免过度设计：

- **字典、配置、日志、文件管理**
- **原因**：强行套用 DDD（如定义 `DictCreatedEvent`, `CreateDictCommand`）只会增加无意义的类文件，没有任何业务收益。

### 3.2 引入 DDD 思想的部分 (业务核心)

对于 **RBAC 核心 (用户/角色/部门)** 等逻辑相对复杂的模块，建议适度引入 DDD 概念：

#### A. 领域事件 (Domain Events) - **强烈建议**

- **场景**：用户密码重置、用户锁定、角色权限变更、部门调整。
- **实现**：利用 `backend-core` 中的 `DomainEvent`。
- **收益**：实现业务解耦。例如 `UserPasswordResetEvent` 发布后，可以由独立的 `AuditLogListener` 记录日志，或 `NotificationListener` 发送通知，而无需在 Service 中硬编码。

#### B. 命令对象 (Commands) - **按需引入**

- **场景**：当写操作涉及复杂的跨聚合校验或多步流程时（如“创建用户”涉及账号唯一性、密码策略、默认角色、部门归属等多重校验）。
- **实现**：使用 `CreateUserCommand` 封装意图。
- **收益**：清晰表达业务意图，配合 `backend-core` 的 `Command` 接口，为未来可能的 CQRS 改造预留空间。

#### C. 查询模型 (Query) - **利用 Jimmer 优势**

- **策略**：读操作不需要强制封装 Query 对象。
- **实现**：充分利用 Jimmer 的 Fetcher 功能，它本身就是一种高效、灵活的 Query 模型。

## 4. 总结

目前不全面使用 DDD 是为了在项目初期实现**快速启动**和**降低复杂度**。未来的架构演进将遵循“实用主义”原则：在 **RBAC 核心业务** 中引入 **领域事件** 以解决耦合问题，而在其他 CRUD 场景中继续保持高效的分层架构。
