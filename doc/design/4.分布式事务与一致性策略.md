## 4.1 设计目标与适用边界

### 4.1.1 设计目标

本章的目标是定义本系统的 **事务处理与数据一致性策略**，在以下约束下达到平衡：

- 保证业务上需要强一致的数据在任意异常场景下“不丢、不乱”（如：用户创建与其基础角色信息）。
- 对大部分跨模块/跨系统场景采用**最终一致性**，最大化吞吐和可用性。
- 减少分布式事务的范围与频率，将复杂度收敛在少数公共基础设施（Outbox、MQ、Seata 入口）。
- 与**性能设计（第 3 章）**和**缓存策略（第 5 章）**协同，不引入不必要的性能瓶颈。

### 4.1.2 一致性需求分级

对系统的业务操作按一致性要求分级：

1. **强一致（本地事务即可）**
   - 单租户内的原子写，如：
     - 创建用户 + 写用户基本信息；
     - 创建角色 + 写角色权限关联；
     - 写资源元数据 + 把资源标记为 ACTIVE。
   - 仅依赖 PostgreSQL 单库内事务即可。
2. **跨模块/跨边界最终一致**
   - 写完业务主表后，异步：
     - 刷新缓存；
     - 发送通知消息；
     - 更新统计报表；
     - 触发后续工作流（LiteFlow/PowerJob）。
   - 使用 Outbox + RocketMQ 模式保证“**要么都不发生，要么 DB 记录一定有对应事件**”。
3. **跨服务强一致（未来阶段，慎用）**
   - 如用户服务与计费服务之间需要对接的场景（此项目当前不涉及金钱交易，可暂不引入）。
   - 若未来出现类似需求，再基于 Seata AT/SAGA 引入真正意义上的分布式事务。

## 4.2 总体策略概述

### 4.2.1 事务优先级排序

从推荐程度高到低：

1. **单库本地事务 + Outbox 事件**（推荐默认方式）
2. **可靠消息 + 最终一致**（通过 MQ 与幂等消费处理跨模块/缓存等）
3. **Seata 全局事务（AT/SAGA）**（仅在确实无法通过上面两种方案实现时使用）

### 4.2.2 Outbox + 事件驱动架构

Outbox 模式是全局的“一致性骨干”：

- 所有需要对外发事件的业务写操作，在本地事务中同时写入业务表和 `t_outbox`。
- 定时/长驻任务从 `t_outbox` 中扫描待处理事件，发布到 RocketMQ。
- 消费方通过业务键与幂等策略确保不会重复处理。

逻辑示意：

```text
[业务写入事务]
   ├─ 写业务表 (e.g. t_user, t_resource_meta)
   └─ 写 t_outbox (status = NEW)

[出站任务]
   ├─ 定期扫描 t_outbox(status = NEW)
   ├─ 发布消息到 MQ（user.events, resource.events 等）
   └─ 更新 t_outbox.status = SENT / RETRY / FAILED

[消费端]
   ├─ 接收 MQ 消息
   ├─ 幂等检查（去重）
   ├─ 执行业务操作（如缓存刷新）
   └─ 打点指标，记录日志
```

## 4.3 Outbox 模式详细设计

### 4.3.1 表结构与字段定义

`t_outbox` 表是统一的消息待发送缓冲区：

```sql
CREATE TABLE t_outbox (
  id             BIGSERIAL PRIMARY KEY,
  tenant_id      BIGINT       NOT NULL,
  aggregate_type VARCHAR(64)  NOT NULL, -- User, Resource, Role 等
  aggregate_id   VARCHAR(64)  NOT NULL, -- 例如 "u123"
  event_type     VARCHAR(64)  NOT NULL, -- UserCreated, RoleChanged 等
  payload_json   JSONB        NOT NULL, -- 通用 JSON 负载
  status         SMALLINT     NOT NULL DEFAULT 0,  -- 0: NEW, 1: SENT, 2: RETRYING, 3: FAILED
  retry_count    INT          NOT NULL DEFAULT 0,
  created_at     TIMESTAMP    NOT NULL DEFAULT now(),
  last_attempt_at TIMESTAMP,
  error_message  TEXT
);

CREATE INDEX idx_outbox_status_created  ON t_outbox (status, created_at);
```

**设计要点：**

- `tenant_id` 方便做租户级出站控制或排查问题。
- `aggregate_type + aggregate_id` 可以作为业务维度的追踪标签。
- `event_type` 与 MQ 的 Topic/Tag 之间通过规则映射。
- `payload_json` 尽量做成紧凑的 JSON，不要过大。
- `status` 与 `retry_count` 支持幂等重试与失败告警。

### 4.3.2 写入 Outbox 的时机

在所有有“对外影响”的本地事务中，例如：

- 用户创建后要刷新缓存/推送 MQ；
- 角色变更后要刷新权限缓存；
- 资源上传后要触发缩略图生成等。

伪代码示例（以创建用户命令为例）：

```java
@GlobalTransactional(name = "createUserCmd", rollbackFor = Exception.class)
public UserId handle(CreateUserCommand cmd) {
    User user = userFactory.create(cmd.tenantId(), cmd.username(), cmd.password());
    userRepository.save(user);

    OutboxEvent evt = OutboxEvent.userCreated(
        tenantId = cmd.tenantId(),
        userId = user.id(),
        username = user.username()
    );
    outboxRepository.save(evt);

    return user.id();
}
```

> 当前阶段可以采用本地事务（Spring @Transactional）即可，不一定需要 Seata 的 @GlobalTransactional。此处使用 @GlobalTransactional 仅作展示，实际可按是否启用 Seata 决定。

### 4.3.3 出站任务实现（调度端）

出站任务通常由 PowerJob 触发（见第 8 章），简化逻辑如下：

```java
public class OutboxDispatchJob implements BasicProcessor {
    private static final int BATCH_SIZE = 200;

    @Override
    public ProcessResult process(TaskContext ctx) throws Exception {
        List<OutboxRecord> records = outboxRepository.fetchNewRecords(BATCH_SIZE);
        for (OutboxRecord r : records) {
            try {
                EventMessage msg = mapper.toMessage(r);
                mqProducer.send(msg); // RocketMQ 发送
                outboxRepository.markSent(r.id());
            } catch (Exception ex) {
                outboxRepository.markRetry(r.id(), ex.getMessage());
            }
        }
        return new ProcessResult(true, "dispatched: " + records.size());
    }
}
```

**注意点：**

- 批次大小控制在合理范围（如 100–500 条）；
- 发送失败时：
  - `retry_count++`；
  - 若达到上限（如 10 次），标记为 `FAILED` 并写告警/通知人工处理；
  - 失败原因记录在 `error_message`，日志中也要记详细。

## 4.4 事件消息与 RocketMQ 集成

此部分与第 11 章“事件总线与异步处理”紧密相关，这里从一致性视角着重说明。

### 4.4.1 Topic 与 EventType 映射

建议为每大领域定义一个 Topic，再用 Tag 或 eventType 区分细类事件。例如：

| 领域       | Topic 名          | 常见 eventType                  |
|------------|-------------------|---------------------------------|
| 用户       | `user.events`     | `UserCreated`, `UserDisabled`   |
| 租户       | `tenant.events`   | `TenantCreated`, `TenantUpdated`|
| 资源       | `resource.events` | `ResourceUploaded`              |
| 权限       | `permission.events` | `RolePermissionChanged`       |
| Excel 导入 | `excel.events`    | `ImportCompleted`               |

Outbox 中 `event_type` 字符串与这些事件类型一一对应，通过统一的映射方法生成 RocketMQ 消息。

### 4.4.2 消费端幂等处理

消费端必须做到：

1. **保证幂等**：
     - 使用业务键（如 `aggregateType + aggregateId + eventType + version`）在本地做去重；
     - 或在 Redis 中使用 `SETNX` 记录该事件已处理。
2. **避免双写冲突**：
     - 对于需要写 DB 的消费逻辑，设计好幂等更新模式：
     - 例如“更新缓存版本号”、“插入若干行但允许重复时忽略”等。

## 4.5 Seata 集成与使用规范

### 4.5.1 当前阶段的定位

- 系统当前是**单体多模块**：
    - 多数业务写操作只涉及一个 PostgreSQL 库和有限 Redis/MQ/RustFS 的组合。
    - 在这种情况下，完全可以用 Spring 本地事务 + Outbox 模式解决绝大多数问题。
- **Seata 的角色**：
  - 作为“未来服务化”的准备工具，主要在以下场景考虑使用：
    1. 出现多个数据库实例需要统一事务时；
    2. 拆分服务后，一个命令需要同时修改多个服务的数据库等资源。

> 当前版本中，Seata 的主要工作是：
>
> - 对数据源进行代理；
> - 提供 @GlobalTransactional 能力；
> - 但仅在少数关键场景开启，不推荐大面积使用。

### 4.5.2 DataSourceProxy 配置

在 Spring Boot 中，将原始数据源交给 Seata 代理：

```java
@Bean
public DataSource dataSource(DataSourceProperties properties) {
    HikariDataSource hikari = properties.initializeDataSourceBuilder()
        .type(HikariDataSource.class)
        .build();
    return new DataSourceProxy(hikari);
}
```

之后，所有 Jimmer 的 EntityManager/数据访问都走这个代理数据源。

### 4.5.3 @GlobalTransactional 的使用规范

**原则**：只有在必须同时保证“多个资源（多库/多服务）的一致写入”时才使用。

示例（未来可能跨服务场景）：

```java
@GlobalTransactional(name = "assignUserAndQuota", rollbackFor = Exception.class)
public void assignUserAndQuota(AssignUserQuotaCommand cmd) {
    userService.assignRole(cmd.userId(), cmd.role());
    quotaService.increaseQuota(cmd.tenantId(), cmd.amount());
}
```

**注意：**

- 事务范围越小越好，避免在 @GlobalTransactional 中做大批量、长时操作。
- 在虚拟线程场景下，须显式传播 XID（RootContext.getXID -> 子线程 RootContext.bind/unbind）。

## 4.6 一致性策略与典型业务场景

本节给出几个关键用例，说明系统如何选择一致性方案。

### 4.6.1 用户创建与权限缓存

**步骤：**

1. Command：CreateUserCommand
2. 本地事务中：
   - 插入 `t_user` 记录；
   - 插入 `t_user_role`（分配默认角色）；
   - 写一条 `UserCreatedEvent` 到 `t_outbox`。
3. 出站任务发布 `UserCreated` 到 `user.events`。
4. 消费端（权限缓存刷新服务）：
   - 在 Redis 中加载/更新该用户的角色/权限缓存；
   - 幂等：若缓存已有且 Version >= 事件 Version，则跳过。

**一致性分析：**

- 强一致部分：用户主数据 + 角色绑定（单库事务保证）。
- 最终一致部分：权限缓存刷新（稍微延迟不影响系统安全，最坏情况短时间内使用旧缓存）。

结论：使用“本地事务 + Outbox + MQ + 幂等缓存刷新”即可。

### 4.6.2 资源上传与后处理（缩略图、索引）

**步骤：**

1. 客户端上传文件 → 后端通过 RustFS 存储内容。
2. 本地事务：
   - 写入 `t_resource_meta`；
   - 写 `ResourceUploadedEvent` 到 `t_outbox`。
3. 出站 → RocketMQ → `resource.events`。
4. 消费端：
   - 任务 1：生成图片缩略图，再写回 RustFS，并更新 `t_resource_meta` 的 metadata 字段；
   - 任务 2（可选）：更新搜索索引（如使用某搜索引擎）。

**一致性分析：**

- 强一致：资源是否存在，以 `t_resource_meta` 为准；内容存储成功则文件可用。
- 最终一致：缩略图、索引这些增强功能失败不影响主流程，可以重试或人工干预。

结论：不需要 Seata，全程使用 Outbox + MQ + 幂等后处理链即可。

## 4.7 缓存与一致性协同（与第 5 章呼应）

与缓存策略结合时，一般流程如下：

1. 写业务 DB 表 + Outbox。
2. 出站任务发布变更事件。
3. 下游消费处理，**负责删除或更新缓存**。

避免在主写事务内频繁操作缓存，减少耦合与延迟。

典型例子：`RolePermissionChangedEvent`：

- 写 DB 更新角色-权限关联；
- 写 Outbox；
- 消费端：
  - 删除/更新 Redis 中 `t:{tenantId}:role:{roleId}:perms`；
  - 若必要，删除所有与该角色绑定用户的权限缓存（或仅标记版本过期）。

## 4.8 错误处理、重试与补偿

### 4.8.1 出站任务失败重试

- 当 RocketMQ 不可用或网络异常等导致发送失败时：
  - `retry_count++`, `status=RETRYING`, `last_attempt_at=now()`；
  - 到达最大重试次数（如 10 次） → `status=FAILED`，同时发送告警。

### 4.8.2 消费端失败策略

- 在消费端：
  - 如果消费逻辑暂时性失败（如 DB/Redis 短暂不可用），由 RocketMQ 的消费重试机制承担；
  - 如果失败是因为数据不合法或严重逻辑错误，则要写入“死信队列”或错误表供人工处理。

### 4.8.3 SAGA 与补偿（预留方案）

对于未来需要跨服务、长链路的一致性要求，可以引入 SAGA 模式：

- 为每个前向操作定义对应的补偿操作；
- 使用 LiteFlow 或 PowerJob 作为编排引擎，维护“执行状态机”和“补偿策略”。

当前版本暂不启用全 SAGA 流程，但在设计上要预留好事件和命令结构，以便后续升级。

## 4.9 与性能、可观测性的联动

- **性能视角**：
    - Outbox + MQ 模式避免了分布式事务在高并发环境下成为瓶颈；
    - 只在确实必要的场景使用 Seata。
- **可观测性视角**：
    - 对 Outbox 写入、出站任务执行、MQ 消费处理等关键点进行指标和日志打点：
      - `outbox.write.count`、`outbox.dispatch.success/fail`、`event.consume.success/fail`、`event.lag.seconds`。
    - 通过 Trace 关联：
      - 从原始 HTTP 请求 → Command → Outbox 记录 → MQ 消息 → 消费端更新缓存/执行后续逻辑，形成闭环。

## 4.10 本章小结

本章确立了整个系统的一致性“骨架”：

1. **本地事务 + Outbox + 事件驱动** 是默认一致性模式。
2. **Seata 全局事务** 只在确实必须的跨库或跨服务强一致场景中启用。
3. 缓存刷新、权限更新、资源后处理等全部通过事件与幂等消费实现**最终一致**。
4. 对所有“重要事务链”提供监控与日志，便于排查异常与调优。

在实际开发中：

- 每个命令（Command）实现时，需要明确：
    - “哪些写操作必须在同一事务内完成”；
    - “哪些后续动作要通过 Outbox 推给 MQ 异步执行”；
    - “是否存在未来需要用 Seata 跨服务的扩展空间”。
