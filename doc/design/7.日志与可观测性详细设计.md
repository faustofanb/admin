## 7.1 设计目标与范围

### 7.1.1 设计目标

1. 保证系统在生产环境具备**强可观测性**：
   - 出问题能快速定位（知道“哪一条链路出了问题”）。
   - 能回答“什么时候慢了”“哪个租户/接口慢”“为什么慢”。
2. 符合“**三大支柱**”：
   - 日志（Logs）：详细事件、错误、审计记录。
   - 指标（Metrics）：聚合数值，用于告警和趋势分析。
   - 链路追踪（Traces）：跨服务/模块的请求链路，包含每一段 span。
3. 面向多租户：
   - 支持按租户维度分析调用量、错误率、性能。
   - 日志、指标、Trace 中统一带 `tenantId` 与 `traceId`。

### 7.1.2 范围

- 日志：
  - 应用日志（业务日志、框架日志） → Promtail → Loki → Grafana。
  - 审计日志（关键业务动作） → PostgreSQL + Loki。
- 指标：
  - 技术指标（JVM、HTTP、DB、Redis、MQ、线程等）。
  - 业务指标（按租户和场景维度：登录次数、Excel 导入量、资源操作等）。
- Trace：
  - HTTP 入口请求。
  - Dubbo RPC 调用。
  - RocketMQ 消费/生产。
  - PowerJob 任务执行。
  - 关键业务流程内部（如上传资源、Excel 导入、LiteFlow 编排）。

## 7.2 日志设计（Logs）

### 7.2.1 日志分类

1. **业务日志（业务模块产生）**
   - 记录用例执行过程中的关键节点、异常信息等。
   - 用于定位业务逻辑错误和行为分析。
2. **技术日志（框架与中间件）**
   - Spring Boot、Jimmer、Seata、Dubbo 等系统组件的运行信息。
   - 一般设置为 INFO 或 WARN，必要时 DEBUG。
3. **审计日志**
   - 记录安全敏感操作：
     - 登录/登出；
     - 权限变更；
     - 资源访问和删除；
     - 调度任务的手工触发等。
4. **访问日志（选做）**
   - HTTP 访问日志（类 nginx access log），可以由反代层记录，也可以在应用层记录摘要。

### 7.2.2 日志格式统一规范

使用 JSON 格式输出，样例：

```json
{
  "ts": "2025-11-20T11:35:24.123Z",
  "level": "INFO",
  "logger": "com.example.app.user.UserCommandHandler",
  "thread": "http-nio-8080-exec-15",
  "traceId": "c1a2b3c4d5e6f7g8",
  "spanId": "ae12ff45",
  "tenantId": "t1",
  "userId": "u123",
  "requestId": "req-20251120-0001",
  "event": "CreateUser",
  "message": "User created successfully",
  "fields": {
    "username": "alice",
    "roleCodes": ["ADMIN"],
    "ip": "10.0.0.1"
  }
}
```

字段约定：

| 字段       | 描述                                           |
|------------|------------------------------------------------|
| ts         | ISO 8601 时间戳（UTC）                         |
| level      | 日志级别（TRACE/DEBUG/INFO/WARN/ERROR）       |
| logger     | 日志记录器名称（类名）                         |
| thread     | 线程名（虚拟线程可另有前缀，如 `vt-...`）     |
| traceId    | Trace ID，来自 OpenTelemetry                  |
| spanId     | 当前 Span ID                                  |
| tenantId   | 当前租户 ID                                   |
| userId     | 当前用户 ID                                   |
| requestId  | 请求 ID（AppContext）                         |
| event      | 业务事件名或动作名                            |
| message    | 简要说明                                      |
| fields     | 动态字段（JSON 对象）                         |

### 7.2.3 日志级别策略

建议：
- 默认等级：`INFO`。
- 开发/测试环境可用 `DEBUG`，生产环境谨慎使用 DEBUG。
- 错误信息必须使用 `ERROR`，且尽量带堆栈。

等级划分示例：
- DEBUG：调试信息，输出频率高但可在生产关闭。
- INFO：关键业务动作成功的简要记录，例如“用户登录成功”、“Excel 导入任务启动”。
- WARN：潜在问题，例如第三方调用超时重试但最终成功。
- ERROR：最终失败场景（未捕获异常、业务处理失败）。

### 7.2.4 敏感信息脱敏与避免

- 禁止直接记录：
  - 密码明文；
  - 完整 Token（JWT 等）；
  - 银行卡号、身份证号等敏感信息（可以播放部分脱敏，如 `****`）。
- 对邮箱、手机号等字段进行脱敏处理：

```java
String maskPhone(String phone) { ... } // 138****1234
String maskEmail(String email) { ... } // u***@mail.com
```

可封装 LogUtil，避免开发自己手写脱敏逻辑。

## 7.3 日志采集与 PLG（Promtail/Loki/Grafana）集成

### 7.3.1 本地输出与文件滚动策略

- 使用 Logback 或 Log4j2 输出到文件：
  - 采用每天滚动（`app.log.yyyy-MM-dd`）；
  - 保留最近 N 天文件。
- 示例 Logback 配置片段（简化）：

```xml
<appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>/var/log/app/app.log</file>
    <encoder class="net.logstash.logback.encoder.LoggingEventCompositeJsonEncoder">
        <!-- JSON 编码 -->
    </encoder>
    <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
        <fileNamePattern>/var/log/app/app.log.%d{yyyy-MM-dd}</fileNamePattern>
        <maxHistory>14</maxHistory>
    </rollingPolicy>
</appender>
```

### 7.3.2 Promtail 配置要点

Promtail 负责从应用容器或主机采集日志并推送给 Loki。示例片段（promtail-config.yaml）：

```yaml
server:
  http_listen_port: 9080
  grpc_listen_port: 0
positions:
  filename: /var/promtail/positions.yaml
clients:
  - url: http://loki:3100/loki/api/v1/push
scrape_configs:
  - job_name: app-logs
    static_configs:
      - targets:
          - localhost
        labels:
          job: app-logs
          app: multi-tenant-backend
          env: prod
          __path__: /var/log/app/app.log*
```

- **labels**：
  - `app`：应用名；
  - `env`：环境（dev/test/staging/prod）；
  - 可在应用端通过日志字段添加 `tenantId`、`traceId` 等，Grafana 中按字段查询。

### 7.3.3 Loki + Grafana 展示

- 在 Loki 中，通过 `app`, `env` 等 label 分片。
- 在 Grafana 中创建 Dashboard：
  - 面板示例：
    - 全局 ERROR 日志数趋势；
    - 按 `tenantId` 聚合的错误日志数量；
    - 特定 Trace 的日志流水。

## 7.4 审计日志设计

> 安全相关细节在第 2 章已有，这里从可观测性角度补充。

### 7.4.1 审计事件范围（复盘）

必须审计：
- 登录/登出。
- 修改用户信息/密码。
- 创建/删除/禁用用户。
- 创建/更新/删除角色及权限。
- 资源上传、下载（特别是下载敏感资源）、删除。
- 手工触发或修改调度任务（如 PowerJob 任务）。

### 7.4.2 审计记录结构

数据库表 `t_audit_action`（已在前文提到，这里重申并补充）：

```sql
CREATE TABLE t_audit_action (
  id          BIGSERIAL PRIMARY KEY,
  tenant_id   BIGINT       NOT NULL,
  user_id     BIGINT,
  action      VARCHAR(128) NOT NULL,
  target_type VARCHAR(64),
  target_id   VARCHAR(128),
  detail      JSONB,
  ip          VARCHAR(64),
  ua          VARCHAR(255),
  ts          TIMESTAMP    NOT NULL DEFAULT now()
);
CREATE INDEX idx_audit_tenant_ts ON t_audit_action(tenant_id, ts);
```

同时，审计事件也写入 Loki（与业务日志统一管道），以便统一检索。

### 7.4.3 审计与可观测的关系

- 安全团队可在 Grafana 中建立审计面板：
  - 登录成功/失败次数（按租户/用户/IP）。
  - 权限变更频率与责任人。
  - 资源访问记录（按类型/等级）。
- 发生安全事件时，可通过审计日志和 Trace 还原整个过程。

## 7.5 指标设计（Metrics）

### 7.5.1 指标体系总览

通过 Micrometer 暴露指标给 Prometheus：
- 技术指标：
  - `jvm.*`、`system.*`、`http.server.requests`、`tomcat.*` 等默认指标。
- 应用指标：
  - 缓存命中/未命中；
  - DB 查询耗时；
  - MQ 发送/消费次数与延迟。
- 业务指标：
  - 登录次数、失败次数；
  - 每租户的接口调用量与 RT；
  - Excel 导入/导出任务数、行数；
  - 资源上传/下载次数。

### 7.5.2 技术指标（Micrometer 默认）

通过依赖：

```xml
<dependency>
  <groupId>io.micrometer</groupId>
  <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
```

在 `application.yml` 启动：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,prometheus
  metrics:
    tags:
      application: multi-tenant-backend
      env: prod
```

Prometheus 将从 `/actuator/prometheus` 拉取指标。

### 7.5.3 应用级自定义指标

示例：记录资源上传耗时与计数：

```java
private final Timer uploadTimer;
private final Counter uploadCounter;

public ResourceMetrics(MeterRegistry registry) {
    this.uploadTimer = Timer.builder("resource_upload_duration_seconds")
        .description("Resource upload latency")
        .publishPercentiles(0.5, 0.95, 0.99)
        .tag("app", "multi-tenant-backend")
        .register(registry);

    this.uploadCounter = Counter.builder("resource_upload_total")
        .description("Total resource upload count")
        .tag("app", "multi-tenant-backend")
        .register(registry);
}

public ResourceId uploadResource(UploadCommand cmd) {
    return uploadTimer.record(() -> {
        ResourceId id = doUpload(cmd);
        uploadCounter.increment();
        return id;
    });
}
```

同理，可为 Excel 导入、Outbox 出站、PowerJob 任务执行等打点。

### 7.5.4 业务指标与多租户维度

建议在一些关键指标中增加 `tenantId` 维度，但要注意：
- 预算卡片原则：不要为每个租户都实时暴露大量指标，避免标签爆炸。
- 可以为“重点租户”添加单独指标标签，或使用 ID 映射方式。

示例（简化处理）：

```java
Counter.builder("login_success_total")
  .tag("tenantId", tenantId)
  .register(registry)
  .increment();
```

或使用采样/聚合方式，避免无限维度膨胀。

## 7.6 Trace 设计（OpenTelemetry）

### 7.6.1 Trace 范围

对以下处理单元都生成/传递 Trace：
- HTTP 请求（Controller 入口）。
- Dubbo RPC 调用（生产/消费端）。
- RocketMQ 消费处理。
- PowerJob 任务执行处理流程。
- 资源模块的 RustFS 调用、Excel 模块的导入流程等关键链路。

### 7.6.2 TraceId 与 AppContext、日志的结合

- 在请求进入时（如 Servlet Filter 或 WebFlux Filter），由 OpenTelemetry SDK 创建/获取 TraceContext。
- 将当前 TraceId 写入 AppContext（用于日志字段、审计记录等）。
- 在日志输出时，通过 MDC 或自定义字段把 TraceId 打印出来。

伪代码示例：

```java
public class TracingFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) {
        Span span = tracer.spanBuilder("http " + request.getMethod() + " " + request.getRequestURI())
                          .setSpanKind(SpanKind.SERVER)
                          .startSpan();
        try (Scope scope = span.makeCurrent()) {
            String traceId = span.getSpanContext().getTraceId();
            TraceIdHolder.set(traceId);
            chain.doFilter(request, response);
        } finally {
            span.end();
            TraceIdHolder.clear();
        }
    }
}
```

### 7.6.3 Dubbo、MQ、PowerJob Trace 传递

- Dubbo：
    使用 Dubbo Filter 将 TraceContext 注入 RPC 调用，并在 Provider 端恢复。
- RocketMQ：
    在消息 Header 中加入 `traceId`，消费端从 Header 中取出并设置当前 Span 上下文。
- PowerJob：
    在 JobContext 参数中携带 `traceId`（如有必要），或在任务启动时新建一个根 Span。

## 7.7 可观测性 Dashboard 与告警

### 7.7.1 Grafana Dashboard 设计

建议至少设计以下几个 Dashboard：

1. **系统总览面板**
   - CPU/内存/GC/MQ 连接数；
   - JVM 线程数、虚拟线程异常情况；
   - 错误率与 RT 总览。
2. **接口性能面板**
   - 按 HTTP 路径统计请求量、失败率、P95/99 延迟；
   - 多租户 Top N 访问量与错误情况。
3. **资源与 Excel 面板**
   - `resource_upload_total`, `resource_download_total`；
   - `excel_import_total`, `excel_import_rows_total`, 错误行比例。
4. **任务调度与 Outbox 面板**
   - Outbox NEW/RETRY/FAILED 数量；
   - PowerJob 任务执行次数、失败率；
   - MQ 消费延迟。
5. **安全与审计面板**
   - 登录成功/失败次数；
   - 权限变更事件数量；
   - 每租户安全相关操作的分布。

### 7.7.2 告警规则

使用 Prometheus Alertmanager 定义告警规则，例如：
- HTTP 5xx 比例 > 1% 持续 5 分钟。
- 任意接口 P95 > 1s 持续 10 分钟。
- Outbox `FAILED` 数量增加、或 `NEW` 长时间不清空。
- 某租户登录失败次数异常升高。

## 7.8 与安全、性能、事务等章节的关系

- 安全（第 2 章）：
    审计日志和安全告警是安全体系的重要组成部分。
- 性能（第 3 章）：
    指标与 Trace 是性能优化的基础数据来源。
- 分布式事务（第 4 章）：
    Outbox 处理、MQ 消费、补偿流程都需要日志和指标来观察一致性。
- 缓存与限流（第 5 章）：
    缓存命中率、Sentinel 限流/熔断次数指标是判断规则合理与否的依据。
- 资源（第 6 章）、Excel、调度等业务模块：
    每个关键业务流程都需要与可观测系统对接，暴露必要的日志、指标、Trace。

## 7.9 本章小结

本章系统性地定义了：

- 日志结构与采集链路（应用 → 文件 → Promtail → Loki → Grafana）。
- 审计日志与安全相关可观测信息。
- 指标体系（技术 + 应用 + 业务）以及 Micrometer + Prometheus 的接入方案。
- Trace 设计（OpenTelemetry），涵盖 HTTP、RPC、MQ、调度与内部流程。
- 可观测 Dashboard 与告警体系。

实际开发时，应当保证：

1. 日志输出统一 JSON 格式，带上 `tenantId`/`userId`/`traceId` 等关键字段。
2. 所有关键业务操作打点指标与 Trace，方便后续运维和问题定位。
3. 将可观测性纳入“验收标准”，不是上线后再补。
