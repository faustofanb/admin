## 12.1 设计目标与场景

### 12.1.1 痛点分析

在复杂的业务场景中（如“企业开户”、“大额订单审批”、“资源上传流水线”），代码往往充斥着大量的 `if-else` 嵌套和冗长的 `Service` 方法。
- **可读性差**：核心流程被细节淹没。
- **维护难**：修改一个步骤可能影响全局。
- **复用性低**：通用的步骤（如“风控检查”）难以在不同流程间复用。

### 12.1.2 选型：LiteFlow

LiteFlow 是一个轻量级、高性能的国产流程编排框架。
- **组件化**：将业务逻辑拆分为微小的组件（Node）。
- **规则化**：通过 EL 表达式（`THEN`, `WHEN`, `SWITCH`）编排流程。
- **热刷新**：支持规则动态调整（存 DB/Nacos）。

## 12.2 规则设计与编排

### 12.2.1 组件定义

所有组件需继承 `NodeComponent`。

```java
@LiteflowComponent("checkUser")
public class CheckUserCmp extends NodeComponent {
    @Override
    public void process() {
        // 获取上下文
        OrderContext ctx = this.getContextBean(OrderContext.class);
        // 业务逻辑
        if (!userService.isActive(ctx.getUserId())) {
            throw new BusinessException("用户状态异常");
        }
    }
}
```

### 12.2.2 EL 表达式编排

**场景：资源上传处理流程**
1. 检查租户配额 (`checkQuota`)
2. 病毒扫描 (`virusScan`)
3. 并行处理：
   - 保存元数据 (`saveMeta`)
   - 生成缩略图 (`genThumbnail`)
4. 发送通知 (`notify`)

**EL 规则**：
```xml
<chain name="resourceUploadChain">
    THEN(
        checkQuota,
        virusScan,
        WHEN(saveMeta, genThumbnail),
        notify
    );
</chain>
```

### 12.2.3 决策路由（SWITCH）

**场景：根据订单金额走不同审批**

```xml
<chain name="orderApproveChain">
    SWITCH(amountSwitch).to(
        managerApprove,  // 金额 < 1000
        directorApprove  // 金额 >= 1000
    );
</chain>
```

## 12.3 上下文管理

LiteFlow 是无状态的，数据通过 `ContextBean` 传递。

### 12.3.1 定义 Context

```java
public class UploadContext {
    private String tenantId;
    private String fileId;
    private long size;
    // ... getters/setters
}
```

### 12.3.2 执行流程

```java
@Resource
private FlowExecutor flowExecutor;

public void upload(UploadReq req) {
    UploadContext ctx = new UploadContext();
    ctx.setTenantId(req.getTenantId());
    // ...
    
    LiteflowResponse response = flowExecutor.execute2Resp("resourceUploadChain", null, ctx);
    
    if (!response.isSuccess()) {
        throw response.getCause();
    }
}
```

## 12.4 异常处理与回滚

### 12.4.1 异常捕获

- 组件抛出异常会中断流程（除非使用 `IGNORE_ERROR`）。
- `response.getCause()` 获取原始异常。

### 12.4.2 事务一致性

LiteFlow 本身不管理事务。
- **小事务**：如果整个 Chain 执行很快，可以在 `execute2Resp` 外层加 `@Transactional`。
- **大事务**：如果包含耗时操作（如 RPC、文件 IO），不建议加全局事务。应采用 **SAGA 模式**（每个组件负责正向操作，并提供补偿操作）。

## 12.5 动态规则与脚本

### 12.5.1 数据库存储规则

将 EL 规则存入数据库 `t_liteflow_rule`，配置 SQL 轮询加载。
- 优点：运营人员/管理员可动态调整流程（如“临时关闭病毒扫描”）。

### 12.5.2 脚本组件（Groovy/QLExpress）

对于经常变的简单逻辑（如“满减计算”），可直接在数据库写 Groovy 脚本，无需发版重启。

```xml
<node id="calcDiscount" type="script" language="groovy">
    <![CDATA[
        def ctx = contextBean
        if (ctx.amount > 100) ctx.discount = 10
    ]]>
</node>
```
