## 11.1 设计目标与场景

### 11.1.1 为什么需要消息驱动

1. **解耦**：核心业务（如“创建订单”）不应直接依赖非核心业务（如“发送短信”、“增加积分”）。
2. **削峰填谷**：应对突发流量（如秒杀、大批量导入）。
3. **最终一致性**：通过 Outbox + MQ 实现跨聚合、跨模块的数据一致性。

### 11.1.2 选型：RocketMQ

相比 Kafka 或 RabbitMQ，选择 RocketMQ 的理由：

- **事务消息**：原生支持事务消息（虽然我们主要用 Outbox，但原生事务也是一种选择）。
- **延迟消息**：支持任意精度的延迟消息（5.0+），适合“订单超时取消”。
- **Tag 过滤**：支持服务端 Tag 过滤，减少消费者网络流量。
- **Java 生态**：纯 Java 开发，易于维护和排查。

## 11.2 消息模型设计

### 11.2.1 Topic 规划

为了避免 Topic 爆炸，建议按**领域模块**划分 Topic，通过 **Tag** 区分具体事件。

**事件分层：**

1. **领域事件 (Domain Event)**：JVM 内部 Spring Event，事务内/后同步执行，解耦模块内逻辑（如：用户注册 -> 发送欢迎邮件）。
2. **集成事件 (Integration Event)**：跨服务边界，走 MQ，用于跨聚合、跨模块通信（如：用户注册 -> 计费系统开户）。

| Topic            | Tag (Event Type) | 说明         |
| :--------------- | :--------------- | :----------- |
| `TOPIC_USER`     | `USER_CREATED`   | 用户创建     |
|                  | `USER_UPDATED`   | 用户更新     |
|                  | `USER_LOCKED`    | 用户锁定     |
| `TOPIC_ORDER`    | `ORDER_CREATED`  | 订单创建     |
|                  | `ORDER_PAID`     | 订单支付     |
|                  | `ORDER_CLOSED`   | 订单关闭     |
| `TOPIC_RESOURCE` | `RES_UPLOADED`   | 资源上传完成 |
|                  | `RES_DELETED`    | 资源删除     |

### 11.2.2 消息体规范

统一使用 JSON 格式，包含标准头信息。

```java
public class DomainEvent<T> {
    private String eventId;    // UUID
    private String traceId;    // OTel TraceId
    private String tenantId;   // 租户上下文
    private String source;     // 产生事件的服务/模块
    private LocalDateTime occurredOn; // 发生时间
    private T data;            // 业务数据 payload
}
```

## 11.3 生产端设计

### 11.3.1 事务消息与 Outbox 模式

为了保证“数据库写入”与“消息发送”的原子性，**严禁**在事务中直接发送普通 MQ 消息。

**方案 A：RocketMQ 事务消息（推荐）**

1. 发送 Half Message。
2. 执行本地事务。
3. 根据事务结果 Commit 或 Rollback 消息。
4. 若无响应，MQ Server 回查事务状态。

**方案 B：Outbox 模式（兜底）**

1. 业务事务内：`INSERT INTO t_outbox_event ...`。
2. 事务提交。
3. 监听数据库变更（CDC）或 定时任务 扫描 `t_outbox_event`。
4. 发送 MQ。
5. 发送成功后删除 Outbox 记录。

### 11.3.2 普通发送（非关键链路）

对于允许丢失的非核心消息（如某些日志、非关键通知），可直接发送。

```java
@Resource
private RocketMQTemplate rocketMQTemplate;

public void sendLog(LogMessage log) {
    rocketMQTemplate.convertAndSend("TOPIC_LOG:INFO", log);
}
```

## 11.4 消费端设计

### 11.4.1 幂等性保证

MQ 可能会重复投递（At Least Once），消费端**必须**实现幂等。

**方案 A：业务主键去重**

- 数据库表有唯一索引（如 `order_id`）。
- 重复插入会报 `DuplicateKeyException`，捕获并忽略。

**方案 B：去重表（Idempotent Table）**

- 创建表 `t_msg_consumed (msg_id, processed_at)`。
- 消费前先查表，消费后插入表。
- 需在同一事务中执行。

### 11.4.2 租户上下文恢复

消费者线程池默认是空的上下文。需在 Listener 中恢复。

```java
@RocketMQMessageListener(topic = "TOPIC_USER", consumerGroup = "CG_ORDER_USER")
public class UserEventListener implements RocketMQListener<DomainEvent<UserDTO>> {

    @Override
    public void onMessage(DomainEvent<UserDTO> event) {
        // 1. 恢复上下文
        AppContext.setTenantId(event.getTenantId());
        TraceContext.setTraceId(event.getTraceId());

        try {
            // 2. 业务处理
            handle(event.getData());
        } finally {
            // 3. 清理
            AppContext.clear();
        }
    }
}
```

### 11.4.3 重试与死信

- **重试**：RocketMQ 默认重试 16 次（时间间隔递增）。
- **死信（DLQ）**：重试失败后进入死信队列。
- **处理**：需建立死信监控告警，人工介入处理死信消息。

## 11.5 延迟消息应用

### 11.5.1 场景：订单超时自动关闭

1. 用户下单，状态 `PENDING`。
2. 发送延迟消息：`Topic=TOPIC_ORDER_DELAY`, `Tag=CHECK_TIMEOUT`, `Delay=30min`。
3. 30 分钟后，消费者收到消息：
   - 查订单状态。
   - 若仍为 `PENDING`，则执行关闭逻辑。
   - 若已支付，则忽略。

## 11.6 本章小结

本章构建了基于 **RocketMQ** 的异步通信体系：

- 采用 **Topic + Tag** 模式管理消息。
- 强制使用 **Outbox 模式** 确保生产端一致性。
- 强调消费端的 **幂等性** 与 **上下文恢复**。
- 利用 **延迟消息** 解决超时类业务需求。
