## 8.1 设计目标与选型

### 8.1.1 设计目标

1. **分布式调度**：支持集群部署，避免单点故障，支持任务分片执行。
2. **多租户隔离**：
   - 能够区分不同租户的任务。
   - 防止恶意租户耗尽调度资源。
3. **高性能与高可靠**：
   - 秒级调度精度。
   - 任务失败重试与告警。
4. **可视化管理**：
   - 提供任务编排、状态监控、日志查看界面（集成 PowerJob Console）。

### 8.1.2 选型：PowerJob

相比 Quartz 或 XXL-JOB，选择 PowerJob 的理由：
- **MapReduce 模型**：原生支持大规模任务分片（如“清理 100 万个过期资源”）。
- **工作流编排**：支持 DAG（有向无环图）任务依赖。
- **轻量级**：无锁设计，性能强劲。
- **容器化友好**：易于在 K8s/Docker 环境部署。

## 8.2 架构集成

### 8.2.1 部署架构

```text
[ PowerJob Server 集群 ]  <-- (HTTP/Akka) --> [ PowerJob Worker (应用实例) ]
        │                                          │
   (元数据 DB)                                (执行业务逻辑)
```

- **Server**：独立部署，负责调度计算、元数据管理。
- **Worker**：集成在我们的 Spring Boot 应用中，负责接收并执行任务。

### 8.2.2 依赖配置

在 `pom.xml` 中引入：

```xml
<dependency>
    <groupId>tech.powerjob</groupId>
    <artifactId>powerjob-worker-spring-boot-starter</artifactId>
    <version>${powerjob.version}</version>
</dependency>
```

配置 `application.yml`：

```yaml
powerjob:
  worker:
    app-name: multi-tenant-backend
    server-address: powerjob-server:7700
    protocol: http
    store-strategy: disk
    max-heavy-task-instances: 64
```

## 8.3 任务处理器设计

### 8.3.1 处理器接口

实现 `BasicProcessor` 接口：

```java
@Component
public class ResourceCleanupProcessor implements BasicProcessor {

    @Resource
    private ResourceService resourceService;

    @Override
    public ProcessResult process(TaskContext context) throws Exception {
        // 获取任务参数（如保留天数）
        String params = context.getJobParams();
        int retentionDays = Integer.parseInt(params);
        
        // 执行业务逻辑
        resourceService.cleanupDeletedResources(retentionDays);
        
        return new ProcessResult(true, "Cleanup success");
    }
}
```

### 8.3.2 多租户任务处理策略

由于 PowerJob 本身不感知业务租户，我们需要在任务参数或逻辑中处理租户上下文。

**策略 A：全量扫描（适合系统级任务）**
- 任务逻辑：遍历所有租户 -> 对每个租户执行清理。
- 优点：配置简单，一个 Job 搞定所有。
- 缺点：租户多时耗时久，单次执行可能超时。

**策略 B：分片处理（MapReduce）**
- `map` 阶段：查询所有 `tenantId`，分发给子任务。
- `reduce` 阶段：汇总结果。
- 优点：并行执行，效率高。

**策略 C：租户级独立任务（适合租户定制任务）**
- 为每个租户动态创建 Job（通过 Open API）。
- 优点：隔离性好，可单独暂停/配置。
- 缺点：Job 数量巨大，管理复杂。

**推荐策略**：
- **系统维护类**（如清理日志、资源）：采用 **策略 B（MapReduce）**。
- **业务报表类**（如月度报表）：采用 **策略 B** 或 **策略 A**。
- **租户自定义定时触发**（如“每天 8 点发邮件”）：采用 **策略 C**（需限制数量）。

### 8.3.3 MapReduce 示例：全租户资源清理

```java
@Component
public class TenantResourceCleanupProcessor implements MapReduceProcessor {

    @Override
    public ProcessResult process(TaskContext context) throws Exception {
        // Root 任务：获取所有租户并分片
        if (isRootTask()) {
            List<String> tenantIds = tenantRepository.findAllIds();
            return map(tenantIds, "CLEANUP_TASK");
        }
        
        // Sub 任务：处理单个租户
        if (context.getSubTask() instanceof String tenantId) {
            // 设置租户上下文
            AppContext.setTenantId(tenantId);
            try {
                resourceService.cleanup(tenantId);
                return new ProcessResult(true, "Tenant " + tenantId + " done");
            } finally {
                AppContext.clear();
            }
        }
        
        return new ProcessResult(false, "Unknown task");
    }
}
```

## 8.4 任务管理与动态调度

### 8.4.1 动态创建任务

对于需要动态生成的任务（如用户设置“明天 10 点提醒我”），调用 PowerJob Server 的 Open API：

```java
@Resource
private PowerJobClient powerJobClient;

public Long createReminderJob(String tenantId, String userId, LocalDateTime time) {
    SaveJobInfoRequest req = new SaveJobInfoRequest();
    req.setJobName("Reminder-" + tenantId + "-" + userId);
    req.setProcessorInfo("com.example.job.ReminderProcessor");
    req.setJobParams(userId);
    req.setTimeExpression(CronUtil.toCron(time));
    // ...
    return powerJobClient.saveJob(req);
}
```

### 8.4.2 任务状态监控

- 通过 PowerJob Console 查看运行日志。
- 集成告警：当任务失败时，PowerJob 支持邮件/钉钉告警，也可对接我们的 PLG 系统（通过日志）。

## 8.5 安全与隔离

### 8.5.1 租户上下文传播

- **关键点**：Worker 线程池中的线程默认没有 `AppContext`。
- **解决方案**：在 `process` 方法入口处，必须根据参数显式设置 `AppContext`，并在 `finally` 块中清理。

### 8.5.2 资源配额

- 防止某个租户的计算密集型任务占满 Worker 线程。
- 方案：
  - 在 `process` 内部加限流（Sentinel）。
  - 监控任务执行时长，超时强制中断（PowerJob 支持超时设置）。

## 8.6 与其它模块的协作

- **与资源模块**：定期清理 `DELETED` 状态的资源文件。
- **与 Excel 模块**：处理长时间运行的异步导入/导出任务（虽然 Excel 更多用 MQ，但定时重试可用 PowerJob）。
- **与日志模块**：任务执行日志需包含 `jobId`, `instanceId`, `tenantId`，便于 Trace。
